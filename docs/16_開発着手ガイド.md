## 1. このガイドの使い方

### 1.1 全体の位置づけ

`08_作業計画_WBS.md`が定める分担は次の通りである。

- **Cowork（調査・設計担当）**: Phase 0（W1〜W8）を完了済み。成果物は`docs/`一式（本書含む）。
- **人＋Claude Code（VSCode）＋GCP（実装担当）**: Phase 1〜5（I-1〜I-9）をこれから進める。各タスクは対応するdocsを参照する。

本書はこの境界線のちょうど上、「調査・設計は終わったが、まだリポジトリも存在しない」という時点から、「I-1〜I-9を回し始められる状態」までを橋渡しする。

### 1.2 人がやること／Claude Codeに任せることの分担方針

作業を大きく4種類に分け、それぞれ適した担い手に割り振る。

| 作業の種類 | 担い手 | 理由 |
|---|---|---|
| **課金・契約・外部サービスの操作**（GCPコンソールでの請求先アカウント設定、GitHubリポジトリの公開/非公開設定、ODPT開発者登録） | **人（必須）** | Claude Codeはこれらの操作権限を持たず、代行もリスクが高い（予算アラート設定・課金有効化は特に人の確認を要する）。4章で手順を具体化する |
| **ファイル・ディレクトリ構造の生成、コード実装、テスト作成** | **Claude Codeに依頼** | `docs/`という正典を読み込ませた上で機械的に実行できる作業。3章・5章の依頼プロンプト例を使う |
| **設計判断が既に確定している定型作業**（pnpm workspace設定、CI設定、feeds.yaml記入等） | **Claude Codeに依頼**（人は結果を確認） | `docs/`に具体的な値・スキーマが既に書かれているため、Claude Codeが正確に再現できる。人は生成結果が docs の記述と一致するかを確認する役に回る |
| **設計判断そのものの変更・未決事項の解消**（`11_アーキテクチャ設計.md` 8章等に列挙された未決事項の決定） | **人（Claude Codeと相談しながら）** | docsが明示的に「未決事項」として切り出している項目は、実装時の状況（実測値、実データ）を見て人が判断すべきものである |

### 1.3 このガイドの読み方

- 2章・3章・4章は**リポジトリが存在する前**にやることであり、原則この順で進める（2→3→4、ただし4章のGCP登録系は2〜3章と並行して進めてよい）。
- 5章は**I-1完了後**、実装フェーズ全体の見取り図として都度参照する。
- 6章・7章は**実装が始まってから**繰り返し参照する運用ガイドである。

## 2. 環境セットアップ手順

### 2.1 バージョン固定方針

`11_アーキテクチャ設計.md` 2章・`08_作業計画_WBS.md`の確定済み設計判断1「Node 22 LTS、pnpm workspace、TypeScript strict、vitest、ESLint(flat config)＋Prettier」に従う。バージョンは以下の方式で固定する。

| 対象 | 固定方法 |
|---|---|
| Node.jsバージョン | リポジトリルートに`.node-version`（値: `22`）を配置し、`package.json`の`engines.node`にも`">=22.0.0 <23.0.0"`相当を明記する |
| pnpmバージョン | `package.json`の`packageManager`フィールド（例: `"packageManager": "pnpm@9.x.x"`、具体的なパッチバージョンは3章のセットアップ時にインストールされた版を記録する）で固定する。Corepack経由の運用を前提とする |
| TypeScriptバージョン | ルートの`devDependencies`で固定（`^5.x`系、`pnpm-lock.yaml`がロックする） |
| その他ツール（vitest, ESLint, Prettier） | 全て`pnpm-lock.yaml`によるロックに委ねる。individualなバージョン指定はモノレポルートの`devDependencies`に集約する（3章のpackage.json雛形を参照） |

### 2.2 インストール手順

以下はWindows環境（PowerShell）を主として記載する。VSCode統合ターミナルでの実行を想定する。

```powershell
# 1. Node.js 22 LTSのインストール（既にNode.jsが入っている場合はバージョン確認から）
node --version
# v22.x.x でなければ https://nodejs.org/ から22 LTS版を入れ直す、
# または nvm-windows / fnm 等のバージョン管理ツールで22系を導入する

# 2. Corepackの有効化（Node.js 16.13+に標準搭載）とpnpmの有効化
corepack enable
corepack prepare pnpm@latest --activate
pnpm --version

# 3. gcloud CLIのインストール確認（4章のGCP初期設定で使用）
gcloud --version
# 未導入の場合は https://cloud.google.com/sdk/docs/install からインストーラーを取得

# 4. Firebase CLIのインストール
pnpm add -g firebase-tools
firebase --version

# 5. GitHub CLI（リポジトリ作成を効率化するため推奨、必須ではない）
gh --version
# 未導入の場合は https://cli.github.com/ からインストール
```

**確認事項**: `node --version`が22系であること、`pnpm --version`が動作すること、`gcloud --version`・`firebase --version`が動作することの4点を、3章に進む前に確認する。

### 2.3 VSCode拡張機能（推奨）

Claude Code拡張機能に加え、以下を推奨する（必須ではない）。

- ESLint拡張機能（flat config対応版）
- Prettier拡張機能
- YAML拡張機能（`feeds.yaml`編集用、`12_データパイプライン設計.md` 2章）

## 3. リポジトリ初期化手順（I-1の具体化）

本章は`08_作業計画_WBS.md`のI-1（「リポジトリ初期化（pnpm monorepo、CI、CLAUDE.md配置）」、参照ドキュメントW8＝本書）を実行可能なコマンド列に落とし込む。`11_アーキテクチャ設計.md` 2章のモノレポ構成（確定済み設計判断2）に従う。

### 3.1 GitHubリポジトリ作成（人の作業）

1. GitHubで新規**公開**リポジトリを作成する（仮名: `norishiro`）。応募条件「成果物はコンテスト期間中、誰でも無償で利用できる状態で公開」（`01_開催概要_募集要項.md`）および`11_アーキテクチャ設計.md` 4章「GitHub Actionsは公開リポジトリでは無料」の前提を満たすため、**必ず公開設定**にする。
2. ライセンスファイルを選ぶ（MIT等、応募規約に反しないもの。具体的な選定は本書のスコープ外、人が判断する）。
3. ローカルにクローンする。

```powershell
gh repo create norishiro --public --clone
cd norishiro
```

（`gh`未導入の場合はGitHub Web UIで作成し、`git clone`で取得する。）

### 3.2 pnpm workspaceの構成

`11_アーキテクチャ設計.md` 2章が定める`packages/*`・`apps/*`構成をディレクトリレベルで先に作る。

```powershell
mkdir packages\types, packages\gtfs, packages\router
mkdir apps\web, apps\mcp, apps\pipeline
mkdir docs
```

ルートに`pnpm-workspace.yaml`を作成する。

```yaml
packages:
  - "packages/*"
  - "apps/*"
```

ルートに`package.json`を作成する（TypeScript strict・vitest・ESLint flat config・Prettierを含む雛形）。

```json
{
  "name": "norishiro",
  "private": true,
  "packageManager": "pnpm@9.0.0",
  "engines": {
    "node": ">=22.0.0 <23.0.0"
  },
  "scripts": {
    "typecheck": "pnpm -r run typecheck",
    "lint": "eslint .",
    "test": "pnpm -r run test",
    "dev": "pnpm --filter apps/web run dev"
  },
  "devDependencies": {
    "typescript": "^5.5.0",
    "vitest": "^2.0.0",
    "eslint": "^9.0.0",
    "prettier": "^3.3.0"
  }
}
```

各`packages/*`・`apps/*`配下に最小限の`package.json`（`name`は`@norishiro/types`のようなスコープ付き命名を推奨）と`tsconfig.json`（ルートの`tsconfig.base.json`をextendし、`strict: true`を明示）を配置する。**`packages/router`は`packages/gtfs`に依存しない**という`11_アーキテクチャ設計.md` 2章の依存方向ルールを、この段階では各パッケージの`package.json`の`dependencies`欄に反映するだけでよい（実装コード自体はI-2・I-3で書く）。

### 3.6 CI（GitHub Actions）の設定

確定済み設計判断5「PR時にtypecheck/lint/testを実行するGitHub Actions」に従う。データパイプラインのワークフロー（`pipeline.yml`、`12_データパイプライン設計.md` 5章）は**別ファイル**として扱い、本節のCIには含めない。

`.github/workflows/ci.yml`を作成する。

```yaml
name: CI

on:
  pull_request:
    branches: [main]

jobs:
  typecheck-lint-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "22"
      - run: corepack enable
      - run: pnpm install --frozen-lockfile
      - run: pnpm run typecheck
      - run: pnpm run lint
      - run: pnpm run test
```

データパイプライン用の`.github/workflows/pipeline.yml`は**I-6のタイミング**で`12_データパイプライン設計.md` 5章の設計（ジョブ分割: `fetch-and-validate`→`normalize-and-build`→`shard-and-credit`→`deploy`）に従って作成する。I-1の時点では作成しない（このタイミングでは`apps/pipeline`自体が未実装のため）。

## 4. GCP初期設定手順

本章は`08_作業計画_WBS.md`の確定済み判断6（GCP初期設定はユーザー作業）を具体化する。**課金に関わる操作は必ず人が行う**（1.2節の分担方針）。Claude Codeに代行させない。

### 4.1 手順概要

1. GCPプロジェクト作成
2. 課金の有効化（請求先アカウントの紐付け）
3. **予算アラート月100円**の設定（`11_アーキテクチャ設計.md` 4章の確定済み防御策）
4. Firebase Hosting初期化
5. Cloud Run API有効化

### 4.2 GCPプロジェクト作成

**コンソール操作**:
1. https://console.cloud.google.com/ にアクセスし、上部のプロジェクト選択メニューから「新しいプロジェクト」を選ぶ
2. プロジェクト名（例: `norishiro-prod`）を入力して作成する

**gcloudコマンド（同等の操作）**:

```powershell
gcloud projects create norishiro-prod --name="norishiro"
gcloud config set project norishiro-prod
```

### 4.3 課金の有効化

**コンソール操作（推奨）**:
1. https://console.cloud.google.com/billing にアクセスする
2. 既存の請求先アカウントがあれば選択、無ければ新規作成（クレジットカード登録が必要）
3. 「お支払いアカウントをリンク」から`norishiro-prod`プロジェクトに請求先アカウントを紐付ける

**確認事項**: Firebase HostingやCloud Runの無料枠内利用であっても、多くのGCPサービスは課金アカウントの紐付け自体を要求する（無料枠を使うためであっても請求先アカウントの登録は必須）。この時点ではまだ何も課金は発生しない。

### 4.4 予算アラート（月100円）の設定 — 無課金防御の中核

`11_アーキテクチャ設計.md` 4章「GCP予算アラート、月100円のしきい値で通知」（確定済み設計判断8）に対応する、最も重要な防御策である。

**コンソール操作**:
1. https://console.cloud.google.com/billing/budgets にアクセスする
2. 「予算を作成」をクリック
3. スコープ: 対象プロジェクト（`norishiro-prod`）を選択
4. 予算額: **100円**（または近似値、GCPコンソールが円建て入力に対応していない場合はUSD相当の少額、例1ドル、を設定し、注記に日本円換算100円目安であることを明記する）
5. アラートしきい値: 50%・90%・100%（GCP既定の複数段階、またはより早期に気付けるよう調整）
6. 通知先: **人が即座に確認できるメールアドレス**を必ず設定する（`11_アーキテクチャ設計.md` 4章「通知先を開発者が即座に確認できるチャネルに設定する」）

**gcloudコマンド（同等の操作、`gcloud billing budgets`サブコマンド使用）**:

```powershell
# 対象の請求先アカウントIDを確認
gcloud billing accounts list

# 予算を作成（BILLING_ACCOUNT_ID は上記コマンドの出力から取得）
gcloud billing budgets create `
  --billing-account=BILLING_ACCOUNT_ID `
  --display-name="norishiro-100yen-alert" `
  --budget-amount=100JPY `
  --threshold-rule=percent=0.5 `
  --threshold-rule=percent=0.9 `
  --threshold-rule=percent=1.0
```

（`gcloud billing budgets create`のフラグ仕様はgcloudのバージョンによって変わることがあるため、実行前に`gcloud billing budgets create --help`で最新のフラグ名を確認する。）

**無課金防御の確認方法**:
- 予算アラート作成直後、GCPコンソールの予算一覧ページ（https://console.cloud.google.com/billing/budgets ）に作成した予算が表示されていることを確認する。
- 通知先メールに「テスト通知」を送る機能がコンソールにあれば、それを使って実際に受信できることを確認する（無ければ、この時点では実際のアラート発火を待つしかない。実装が進みCloud Run等の課金対象リソースが稼働し始めた段階で、コンソールの「お支払い」ダッシュボードから現在の課金額が0円または僅少であることを定期的に目視確認する運用に切り替える）。
- **このアラートには課金を自動的に止める強制力はない**（`11_アーキテクチャ設計.md` 4章の注記通り）。アラート発火後は7章のトラブルシュート、および`14_MCPサーバーAPI仕様.md` 6章のmax-instances見直し等を人が手動で適用する。

### 4.5 Firebase Hosting初期化

**コンソール操作**:
1. https://console.firebase.google.com/ にアクセスし、「プロジェクトを追加」で既存のGCPプロジェクト（`norishiro-prod`）を選択してFirebaseを有効化する

**Firebase CLIでの初期化（リポジトリルートで実行）**:

```powershell
firebase login
firebase init hosting
# プロンプトで既存プロジェクト norishiro-prod を選択
```

## 6. 開発の進め方

### 6.1 フィクスチャ駆動TDDの原則

`08_作業計画_WBS.md`のI-2が明示する通り、瑞穂町データ（`packages/gtfs/tests/fixtures/mizuho/`の12ファイル）を**フィクスチャの正**として開発を進める。具体的な進め方は以下の順とする。

1. **テストを先に落とす**: `docs/10_GTFS-Flex実装仕様.md` 5章（T-P01〜T-P09, T-R01〜T-R08, T-V01〜T-V03）、`docs/13_ルーティングエンジン設計.md` 10章（T-RAPTOR-*, T-FLEX-*, T-PROP-*, T-API-*）は、実装より先にテストコード（vitest）として書く。この時点ではテストは全て失敗する（Red）。
2. **実装する**: テストが通る最小限の実装を書く（Green）。
3. **リファクタリングする**: 9章の性能設計（TypedArray化等）はこの段階で適用する（Refactor）。

これはI-2（パーサ）・I-3（ルーティング）の両方に適用される。**新機能はテスト先行**というCLAUDE.md（成果物2）の方針とも一致する。

### 6.2 docs/13のテストケースを最初に落とす、の具体的な意味

「テストケースを最初に落とす」とは、`docs/13_ルーティングエンジン設計.md` 10章が列挙する19個のテストケース（T-RAPTOR-01〜05、T-FLEX-01〜05、T-PROP-01〜05、T-API-01〜04、および10.1節のT-R-DUR-00〜03）を、`packages/router`の実装コードを書く**前に**、`packages/router/tests/`配下にvitestのテストファイルとして書き切ることを指す。特に10.1節の実測値（殿ケ谷会館〜みずほ病院間2005.5m、推定所要時間759.4秒等）は具体的な数値がdocsに明記されているため、この数値をそのままアサーションに使える。テストが「実装が存在しないためコンパイルエラー」になる段階を経てから実装に着手することで、API契約（8章の型定義）を先に固定してから内部実装を進められる。

### 6.3 性能計測の習慣

`docs/13_ルーティングエンジン設計.md` 9章が定める性能目標（都道府県シャード規模で1クエリ100ms以内、キャッシュ済みシャードでの計測）は、実装が進むにつれて「実測してみたら想定と違う」ことが起こりうる領域として9.5節・11.2節（U-2）が明示的に警告している。以下を習慣化する。

- I-3完了時点で、瑞穂町規模のフィクスチャでの計測結果を記録する（このスケールでは目標達成が容易なはずだが、ベースラインとして記録する）。
- I-6でのシャード実測サイズ確定後（`docs/12` 8.2節U-1）、都道府県規模の合成フィクスチャまたは実データでの計測を行い、100ms目標との差分を確認する。
- 計測結果が目標を外れる場合、`docs/13_ルーティングエンジン設計.md` 11章の代替案（A-3: 空間インデックス限定によるFlexスキャン最適化等）を検討する。**性能問題への対応は、まず該当docsの11章・9章に既に列挙されている対策を確認してから、新しい最適化を考える**（既にdocsが検討済みの選択肢を再発明しない）。

## 7. トラブルシュート想定

実装中に発生しうる、docsが既に想定している逸脱パターンへの対処法を整理する。**新しい対処法を考える前に、まず該当するdocsのセクションを確認する**という原則で読む。

### 7.1 シャードサイズ超過に遭遇した場合

**症状**: `apps/pipeline`のshard生成ステージ（`docs/12_データパイプライン設計.md` 3.5節）で、都道府県シャードが3MB超、またはバックボーンが2MB超になる。

**対処**:
1. `docs/12` 3.5節の確定済みエラーポリシーを確認する: 「超過した場合はビルドを失敗させず**警告としてジョブサマリに記録**し、超過したシャードについては前回成功版を維持するかそのまま採用するかを運用判断とする（v1では警告のみで採用を継続する方針を既定とする）」。**まずビルドを失敗させない**ことを優先する。
2. `docs/12` 8.2節の未決事項1「シャード実測サイズによるバイナリ化判断」・未決事項3「シャードサイズ超過時の運用ルール確定」を確認する。実測が判明した段階で、確定済み設計判断4（v1圧縮JSON、性能不足が判明した時点でバイナリ化を検討する）の発火条件に当たるかどうかを人が判断する。
3. 応急処置として都道府県分割の粒度そのものを見直す（例: 東京都のように大規模事業者フィードを内包するシャードをさらに分割する）ことは`11_アーキテクチャ設計.md` 8章未決事項1「地域シャードの最終分割粒度」に委ねられているため、独自に粒度変更を実装する前にこの未決事項を人と相談する。

### 7.2 Flexデータの仕様逸脱に遭遇した場合 → docs/10の寛容パーサ方針へ

**症状**: 9自治体分のチャレンジ限定Flexデータ（I-8）、または新たに追加した固定路線フィード（I-6）を取り込む際に、瑞穂町データでは発生しなかった構造（列の欠落、値の逸脱、BOM付きファイル、Shift_JISエンコーディング等）に遭遇し、パーサが例外を投げる、または誤った結果を出す。

**対処**:
1. **まず`docs/10_GTFS-Flex実装仕様.md` 4章「寛容パーサ要件」を確認する**。4.2節「想定すべき揺れ」は、BOM付きUTF-8、列順の違い、空ファイル・存在しないファイル、改行コードの混在、CSVエスケープ、全角文字混在、緯度経度桁数不揃い、Shift_JISフォールバック等、実際に多くの逸脱パターンを列挙済みである。遭遇した逸脱がこのリストに含まれる場合、**既にパーサが対応すべき既知のパターンとして設計されている**ため、実装（`packages/gtfs`のRaw層読み込み処理）がこの方針通りに作られているかを確認する。
2. 4.3節「TypeScriptでの型定義方針」のフォールバックルール1〜5（trip_id欠損→行破棄、locationRef3つとも欠損→unresolved、時刻情報両方欠損→終日利用可能扱い、pickup/drop_off_type欠損→デフォルト値、外部キー不整合→警告ログ）を確認し、遭遇したケースがこのフォールバックルールでカバーされているかを確認する。カバーされていない新しいパターンであれば、既存のフォールバックルールの「優先順位順」の考え方（安全側に倒す、行を無視しない）に沿って新しいルールを追加する。
3. **3パターン全対応の前提を忘れない**: `docs/10` 1.3節が明示する通り、v1のRAPTOR統合（`docs/13`の4章）は②location_group型のみを対象とする（NS-5、非スコープ）。9自治体データに①GeoJSONゾーン型や③従来stopの時間窓型が混在していた場合、パーサ（`packages/gtfs`）側は受け取れる設計になっているが、ルーティング統合（`packages/router`）側は瑞穂町と同じlocation_group型のみ対応という前提が崩れる。この場合は「パーサは正規化できるが、ルーティングは対応外」という状態を明示的にエラー・警告として扱い、`11_アーキテクチャ設計.md` 8章未決事項5「9自治体分のチャレンジ限定Flexデータの実際のデータパターン」の解消として人に報告する（黙って無視しない）。
4. 新しい逸脱パターンをテストケースとして追加する場合、`docs/10` 5.1節のT-P07〜T-P09のような「改変フィクスチャによる寛容性検証テスト」の形式を踏襲する（実データを直接改変せず、フィクスチャのコピーを改変してテスト専用データを作る）。

### 7.3 その他の未決事項に遭遇した場合

各docsの末尾に「未決事項リスト」（`docs/11` 8章、`docs/12` 8.2節、`docs/13` 11.2節、`docs/14` 9.2節、`docs/15` 10章相当）が整理されている。実装中に判断が必要な場面に遭遇したら、まず該当するdocsの未決事項リストに同じ論点が既に列挙されていないかを確認する。列挙されている場合、その未決事項が「決定予定」としている時期・条件（例: 「9自治体分のチャレンジ限定データ入手後」）に沿って、拙速に独自判断で仕様を確定させず、人と相談する。
