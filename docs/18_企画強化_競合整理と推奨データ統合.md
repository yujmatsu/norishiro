# 企画強化: 競合整理と推奨データ統合（2026-07-02）

背景: ユーザーから2つの懸念が提起された。
①現行案がデマンド交通に特化しすぎでは。NTT西日本のデマンド交通ソリューションの方がシステムとして優れているのでは
②募集内容は「様々なデータを最大限に活用」を求めており、強く推奨されるほこナビ/LINKS/PLATEAUを現行設計は使っていない

## 1. NTT西日本のサービスの正体（実査結果）

https://business.ntt-west.co.jp/smart10x/solution/040.html を確認した結果:

- 中身は**順風路株式会社「コンビニクル」の販売取次**（ページ内に明記）
- 機能: 予約受付→AI配車計画の自動生成→オペレーター/ドライバー/利用者への通知。全国60か所以上で稼働（2022年時点）
- 顧客: 自治体・運行事業者（B2G SaaS）

**重要な事実**: 順風路は本チャレンジのデマンド交通協力3社の1つであり、川越市「かわまる」のGTFS-Flexデータ提供元。つまりコンビニクルは**ノリシロが読むオープンデータを生成する側のシステム**である。

## 2. 競合ではなく補完 — レイヤーで整理

```
[発見・計画レイヤー]  ← ★ノリシロはここ。全国横断・無料・オープン
   「そもそもどんな移動手段があるか」「固定路線と繋いでどう行くか」
        ↑ GTFS-Flex / GTFS（オープンデータ）
[データ公開レイヤー]   ODPTセンター・GTFSリポジトリ
        ↑
[運行管理レイヤー]    コンビニクル(順風路/NTT西)・チョイソコ(アイシン)・
                     SWAT・MONET・ドコモAI運行バス 等 ← 予約受付・配車最適化
```

- 運行管理システムは**契約した自治体の域内で閉じる**。コンビニクル導入60地域のサービスは、隣町の住民・遠方の家族・観光客・乗換案内アプリからは「存在自体が見えない」
- ノリシロはこの「見えない」を解く発見・計画レイヤーであり、運行システムとは役割が別。むしろ運行システムの普及（=GTFS-Flex公開元の増加）はノリシロの価値を上げる
- 「コンビニクルの方がシステムとして優れている」は比較軸が異なる。配車最適化はあちらの仕事、横断的な経路計画と情報アクセスはこちらの仕事。**同じ土俵で戦わない設計こそが強み**（審査でも「既存商用サービスとの違いは」と必ず聞かれる。この整理が回答になる）
- 将来接続点: booking_url/booking_ruleを介して「ノリシロで発見→運行システムで予約」のディープリンク連携が自然な発展形として語れる

## 3. ただし懸念②は妥当 — 現行設計のギャップ

募集内容の再読により確認したギャップ:

| 募集内容の要求 | 現行設計 | 評価 |
|---|---|---|
| 公共交通オープンデータの活用（必須） | ODPTセンター＋GTFSリポジトリ | ✅充足 |
| ほこナビ/LINKS/PLATEAU（**強く推奨**） | **未使用** | ❌ギャップ |
| パートナーデータ（推奨: e-Stat・気象庁・警察庁等） | 未使用 | ❌ギャップ |
| 「様々なデータを最大限に活用」「地方での課題解決」 | 交通データのみで完結 | ⚠️弱い |

審査基準「オープンデータ活用におけるインパクト」で、推奨データ未使用は明確な減点要素になり得る（docs/02: 新公開データの即活用が評価される傾向とも整合）。

## 4. 強化案 — 「デマンド交通検索」から「生活アクセス案内」へ

コンセプトのリフレーム: **「その町で、病院・買い物・駅まで、どうやって行けるか」に答えるサービス**。デマンド交通は手段の一つに位置づけ直す（技術的な背骨=Flex対応ルーティングは不変。見せ方と入口を変える）。

### 統合するデータと機能（優先順）

| # | データ | 追加機能 | 審査への効き | 実装コスト |
|---|---|---|---|---|
| A | **ほこナビ 歩行空間NW**（強く推奨） | 徒歩レッグをバリアフリー対応に: 段差・勾配を避ける経路、レッグ単位のバリア情報表示、「階段を避ける」オプション | 高齢者ペルソナと完全整合。別所審査員（歩行者ナビ・障がい者支援）に直球。「強く推奨」消化 | 中: 歩行NWの取込とルーター徒歩エッジの差替（提供地域のみ、他はHaversineフォールバック） |
| B | **e-Stat 高齢者人口メッシュ**（推奨） | 到達圏モードに重畳: 「デマンド交通の有無で、通院可能な高齢者が◯人変わる」の定量表示 | EBPM文脈で国交省・LINKS賞に効く。審査プレゼンの山場が数字で語れる | 低: 静的メッシュのJOINと表示のみ |
| C | **医療機関・生活施設POI**（LINKS/G空間等、所在要確認） | 目的地プリセット「病院」「薬局」「役場」「スーパー」— 住所入力不要の目的地選択UI | 「通院・買い物」ユースケースが名実ともに完成。UI/UXの分かりやすさ向上 | 低〜中: POIデータの所在確認が先決 |
| D | **PLATEAU**（強く推奨） | 到達圏の3D可視化（建物の高さ付き都市モデル上に到達圏をアニメーション） | デモ映え・「強く推奨」消化。ただし計算の本質には使わないため「飾り」と見られるリスクあり | 中: 表示層のみ。P2（12月の磨き込み）扱い |
| E | 気象庁（推奨） | 徒歩レッグの雨天注意・予約リマインドに天気添付 | 小 | 低。P2扱い |

- A〜Cを**採用**、Dは12月の磨き込みで余力があれば、Eは任意 — という優先付けを推奨
- 警察庁事故データは本コンセプトとの接点が薄く、無理に使わない（取ってつけた活用は逆効果）

### リフレーム後のピッチ（一文）

「ノリシロは、乗換アプリに出てこないデマンド交通と、段差を避けた歩行経路までつないで、交通空白地帯の『病院に行けない・買い物に行けない』に答える生活アクセス案内です。」

## 5. 審査基準への効き直し

## 8. 交通系MCPの先行事例調査（2026-07-02 Web検索実査）

「ノリシロのMCPは世に出ていないものか」の確認結果。

### 既に存在するもの

| 先行事例 | 内容 | ノリシロとの違い |
|---|---|---|
| **駅すぱあと API MCPサーバー**（ヴァル研究所、2026-01-15リリース、2026-03-26ダイヤ探索追加） | 経路探索・運賃計算・駅情報・ダイヤ探索をMCPツールとして提供。交通費精算などのAIエージェント連携が主用途 | **商用API（契約・APIキー前提）**。固定路線（鉄道中心）のみ。デマンド交通・GTFS-Flex非対応。オープンデータ非依存 |
| 海外の汎用GTFS MCP（jdamcd/gtfs-mcp、caltrain-mcp等のOSS） | GTFS/GTFS-RTフィードを読み込み時刻表・出発案内・車両位置をクエリ | **データ照会が主で、経路計画（ジャーニープランニング）を持たないものが大半**。GTFS-Flex非対応。日本特化でない |
| （参考）Google Maps系MCP | 汎用地図・経路 | デマンド交通・予約ルールの概念なし |

Anthropicコネクタレジストリ・国内検索とも、ODPTオープンデータやGTFS-Flexを扱うMCPは発見できず（2026-07-02時点）。

### 結論 — 何が新しくて何が新しくないか

- **「公共交通×MCP」自体は既出**。「日本初の公共交通MCP」という主張は不可（駅すぱあとが先行）
- **未出と確認できた組み合わせ**（ノリシロの主張可能な新規性）:
  1. **GTFS-Flex／デマンド交通対応**のMCP（予約ルール・締切判定を含む経路計画ツール）
  2. **オープンデータのみで構築され、誰でも無料・契約不要**で使える日本の公共交通MCP
  3. 到達圏（isochrone）・データ出典一覧までツール化した交通MCP
- **駅すぱあとMCPの存在はむしろ追い風**として語れる: 「AIエージェントから交通情報を引く需要は商用側で実証済み。ノリシロはそれをオープンデータ×デマンド交通×無料で民主化する」
- 審査ヒアリング対策: 「駅すぱあとMCPとの違いは？」への回答を上表で準備済み。応募資料・ピッチでは「デマンド交通対応の経路検索MCPとしては初（当社調べ、2026-07時点）」という正確な表現を使う
