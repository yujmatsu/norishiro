### 1.2 データ形式（歩行空間ネットワークデータ整備仕様 2024年7月版）

`https://www.mlit.go.jp/sogoseisaku/soukou/content/001757259.pdf`（国土交通省 政策統括官付、2024年7月）を全文確認。

**構造**: 「リンク」（歩行経路の線）と「ノード」（結節点）の2要素からなるグラフ構造。

**データの階層**:
- 第1層データ: 移動支援サービスに最低限必要な項目（リンクID、起点/終点ノードID、リンク延長、ランク区分、ランク区分設定方法、更新日）
- 第2層データ: 高度化のための選択項目（経路の構造・種別、方向性、幅員、縦断勾配、段差、信号機、視覚障害者誘導用ブロック、エレベーター、屋根の有無、供用時間・曜日制限、通行制限等、全54項目）
- 第3層データ: 地域独自の追加項目（本仕様では定義しない）

**リンクの主要属性**（表3.2、全54項目中の要点）:

| 属性 | 内容 |
|---|---|
| ランク区分（rank） | 幅員・縦断勾配・段差をそれぞれS/A/B/C/Z/Xの6段階でランク化し3文字で表現（例: SAB） |
| 幅員（width） | 1:1.0m未満／2:1.0〜2.0m未満／3:2.0〜3.0m未満／4:3.0m以上／99:不明 |
| 縦断勾配（vtcl_slope） | 1:0%／2:0〜5%以下／3-4:5〜8%以下（上り/下り）／5-6:8〜18%以下／7-8:18%超／99:不明 |
| 段差（lev_diff） | 1:0cm／2:0〜2cm以下／3:2〜5cm以下／4:5〜10cm以下／5:10cm超／99:不明 |
| 経路の構造（rt_struct） | 車道歩道分離あり/なし、横断歩道、地下通路、歩道橋、施設内通路等（コード1〜8、99） |
| 経路の種別（route_type） | 動く歩道、踏切、エレベーター、エスカレーター、階段、スロープ等 |
| 方向性、通行制限、供用時間/曜日制限 | 一方通行・時間制限のある通路（駅構内等）を表現可能 |

**ノードの主要属性**（表3.8）: ノードID、緯度、経度、階層数（屋外=0、屋内は階層順、中間階は0.5刻み）、標高値、施設内外区分（施設外/境界/施設内）、接続リンクID。

**ファイル形式**: CSV（`link.csv`/`node.csv`）、シェープファイル（`link.shp`等4ファイル構成）、GeoJSON（`link.geojson`/`node.geojson`）、XML（GML）。実データセット（赤羽駅周辺で確認）は**CSV+GeoJSONの4ファイル構成**（link.csv, link.geojson, node.csv, node.geojson）で公開されている。文字コードはUTF-8基本。座標系はJGD2011（世界測地系）。

### 1.4 経路検索の徒歩レッグに使う際の実装上の注意

1. **リンクの接続性**: リンクは道路の両側に配置されることが多く（表3.1の分類）、市区町村単位で整備・公開される想定（3.4節「データを整備したエリア（市区町村）名を設定したフォルダに格納」）。**複数市区町村のデータを結合する場合、境界ノードの座標一致を前提にした名寄せが必要**（整備仕様3.2.3項6）で「市区町村の境界にノードを配置する」と明記されており、境界での接続は仕様上保証されているが、実データでの座掛の厳密一致は要検証）。
2. **鉄道駅との接点**: 整備仕様は「屋外と屋内・地下との境界点」「大規模施設・公共施設等の出入口付近へのノード配置」を推奨事項として明記（3.2.3項4・7）。ただし**必須ではなく「配置しておくと良い」という推奨表現**であり、駅施設内通路データ（国土地理院「階層別屋内地理空間情報データ仕様書」準拠）と歩行空間ネットワークデータが常にリンクで直結している保証はない。駅接続を前提にするなら、実データ取得後に「駅出入口ノード」の有無を個別確認する必要がある。
3. **本プロジェクトでの使い所**: 1.1の結果よりMVP圏・9自治体圏では使えないため、**当面は東京都心部のデモ用ショーケース地域（例: 新宿駅周辺、赤羽駅周辺）でのみ「歩行NWグラフを使った徒歩レッグ」を実演し、それ以外の地域はHaversineフォールバックのまま**という運用が現実的（docs/18のA案が想定する「提供地域のみ差替、他はHaversineフォールバック」の想定通り。ただし提供地域がMVP圏と重ならないため、当初想定していた「瑞穂町の高齢者ペルソナに歩行NWで応える」というピッチの説得力は薄れる。デモ地域を分けて「歩行NW対応地域では段差回避ができる」という機能ショーケースとして提示するのが妥当）。

## 2. e-Stat 高齢者人口メッシュ

### 2.1 データ入手方法

`https://www.e-stat.go.jp/` および配下ページを確認。入手経路は2つ:

| 経路 | URL | 概要 |
|---|---|---|
| ① 統計地理情報システム（jSTAT MAP）経由の一括ダウンロード | `https://www.e-stat.go.jp/gis/statmap-search?type=1`（統計データ）、`type=2`（境界データ） | ブラウザで政府統計名・地域を選択してZIP等をダウンロード。API不要でユーザ登録も不要（画面上は登録不要のダウンロード動線に見えるが、詳細な絞込UIは要ログインの可能性あり、未確認） |
| ② API（バージョン3.0） | `https://www.e-stat.go.jp/api/` | プログラムからXML/JSON/CSVで取得。**ユーザ登録＋アプリケーションID(appId)取得が必須** |

### 2.2 API利用の具体的な手順・パラメータ

「利用ガイド」（`api-info/api-guide`）に記載の手順:
1. e-Statユーザ登録
2. マイページからアプリケーションID（appId）を発行（用途に応じて複数取得可、URL欄はローカル用途なら`http://test.localhost/`等でよい）
3. API仕様（バージョン3.0、`api.e-stat.go.jp/rest/3.0/app/...`）に従い開発
4. 公開時はクレジット表示

**小地域・地域メッシュデータ取得の要点**（API仕様書より抽出）:
- 統計表情報取得の`searchKind`パラメータに **`2`（小地域・地域メッシュ）** を指定する
- 小地域・地域メッシュデータは**バージョン3.0でのみ取得可能**（3.0で新規対応、2.1以前は非対応）
- 提供データ一覧ページ（`/stat-search/database`）には小地域・地域メッシュは表示されない。取得にはAPI経由が前提
- 全APIで`appId`パラメータが必須（`〇`必須マーク）
- レスポンス形式: XML/JSON/JSONP/CSV。gzip圧縮対応、CORS対応、HTTPS対応

### 2.3 メッシュコード体系の要点

総務省統計局の地域メッシュ統計解説ページ階層（`https://www.stat.go.jp/data/mesh/`）の構成を確認（本文はShift_JIS文字コードのため文字化けが生じたが、ページ構造・リンク一覧からメッシュ種別が「概要」「作成」「都道府県コード一覧」「用語の解説」の各解説ページに分かれていることを確認）。

**一般に知られているメッシュコード体系**（本調査で直接全文精読はできなかったため、ページ階層・タイトルからの推定を含む。詳細な桁数定義は未確認）:
- 1次メッシュ（約80km角）→ 2次メッシュ（約10km角）→ 3次メッシュ（約1km角、基準メッシュ）→ 4次メッシュ（約500m角、3次の2分割）→ 5次メッシュ（約250m角、3次の4分割）
- コードは緯度・経度から機械的に算出される数値コード（1次メッシュコードの後ろに2次・3次の分割コードを連結する桁構造）
- 年齢階級別人口は国勢調査（小地域・地域メッシュ統計）に含まれ、5歳階級区分等で「65歳以上人口」等の高齢者関連集計が可能な見込み（**具体的な集計表ID・階級区分の確認は今回未実施、未確認**）

### 2.4 ライセンス

e-Stat上の統計データ全般は政府統計であり、多くは**CC BY 4.0相当**（政府標準利用規約に基づく二次利用可）。ただし小地域・地域メッシュ統計固有の利用条件文言は今回未確認（**未確認事項**）。

### 2.5 実装への示唆

- API経由でのメッシュ統計取得には**ユーザ登録+appId取得という「人による事前作業」が発生する**（docs/17のU-系課題と同種）。パイプライン（GitHub Actions）に組み込む場合、`ESTAT_APP_ID`のようなSecretを別途用意する必要がある。
- メッシュ統計は「都道府県・市区町村単位で毎回変わるものではない静的データ」（国勢調査は5年ごと）のため、**パイプラインでの都度取得ではなく、事前に1回ダウンロードしてリポジトリまたはビルド時キャッシュに固定する運用**が合理的（feeds.yamlの他のGTFSフィードのような「毎回fetch」とは性質が異なる）。
- メッシュポリゴンとFlex/固定路線データの空間結合（どのメッシュがどの停留所の到達圏に含まれるか）はビルド時の後処理として実装し、シャードには結合済みの集計値（メッシュ単位の高齢者人口）を埋め込む設計が良い。

### 3.2 推奨取得元と理由

**医療機関: 国土数値情報 医療機関データ（P04）を推奨**。
- 全国一括データが単一ファイルで入手可能（都道府県別に47回アクセスする必要がない）
- ライセンスがCC BY 4.0で完全に自由（商用可、出典表記のみ）
- 診療科目・救急告示病院・災害拠点病院等の属性が「病院を目的地として提示する」UIに直接使える
- 更新頻度は数年おき（2010→2014→2020年度版という更新履歴）だが、位置情報の性質上、高頻度更新の必要性は低い

**高齢者福祉施設（老人ホーム等）: 国土数値情報 福祉施設データ（P14）の福祉施設大分類「02:老人福祉施設」を推奨**。
- ただしライセンスが都道府県・政令指定都市・中核市ごとに例外がある「CC BY 4.0（一部制限）」のため、**MVP圏（西多摩地域の該当自治体）・9自治体圏それぞれの利用条件をExcel一覧で個別確認する追加工数が発生する**（今回は未実施、要確認事項として記録）

**役所**: 国土数値情報「市町村役場等及び公的集会施設（P05, CC BY 4.0）」または「市区町村役場（P34, 非商用）」が候補。P05はCC BY 4.0で自由度が高く推奨。

**スーパー等商業施設**: 適切な政府オープンデータは見つからなかったため「無い」と明記する。**OSM（OpenStreetMap）を補助的に使う場合はODbL（Open Database License）ライセンスに注意**が必要（共有時の同ライセンス継承義務があり、CC BYやCC0とは互換性の扱いが異なる。本プロジェクトのオープンデータ活用方針と矛盾しないか事前確認が必要）。

### 3.3 実装上の注意

- 国土数値情報のダウンロードは都道府県単位のZIPファイル配布（GMLベース）であり、GitHub Actions等のパイプラインで自動取得する場合、都道府県コード→ZIP URLのマッピングをfeeds.yaml的な設定に落とし込む必要がある
- 更新頻度が低い（年度更新）ため、GTFSのような高頻度fetchは不要。年1回程度の手動または低頻度スケジュールでの更新が適切

## 4. まとめ

### 4.1 A/B/Cの「使えるか」判定

| データ | 使えるか | 判定根拠 |
|---|---|---|
| A. ほこナビ歩行空間NW | **MVP圏・9自治体圏では使えない。東京都心部等の限定地域でのみ使える** | 全31データセットがMVP圏・9自治体圏を1件もカバーしない（1.1節）。デモ用ショーケース地域として東京都心部の整備地域（新宿・赤羽等）を使う運用に切替が必要 |
| B. e-Stat高齢者人口メッシュ | **使える（要事前準備）** | 全国どこでも取得可能な統計データ。ただしAPI利用にはユーザ登録・appId取得という人的作業が先行する。メッシュコード体系の詳細（4次/5次メッシュの正確な桁定義）は未確認のまま実装着手すると手戻りリスクがある |
| C. 医療機関・生活施設POI | **医療機関は使える（国土数値情報P04推奨）。福祉施設は使えるが利用条件確認の追加工数あり。商業施設は使えるデータが無い** | P04は全国CC BY 4.0で障壁なし。P14（福祉施設）は自治体ごとの利用条件例外があり要個別確認。商業施設は政府オープンデータが存在せず、OSM併用時はODbLライセンスの取扱いに注意 |

### 4.2 パイプライン取込の追加工数所感

- **A（ほこナビ）**: 中程度の工数増。CSVの列名・コード体系は仕様書で明確に定義されているためパーサ実装自体は素直だが、（i）市区町村境界でのノード名寄せ、（ii）駅出入口ノードとの接続確認、（iii）整備地域が更新され続けている（2026年3月にも新規追加あり）ため定期的な再取得が要る、という3点で運用工数がかかる。かつMVP圏では使えないため「全国対応の基盤は作るが、当面はデモ用の1〜2地域のみ実データを組み込む」という限定スコープが現実的。
- **B（e-Stat メッシュ）**: 低〜中程度の工数。ユーザ登録・appId取得（人的作業、1回のみ）を終えれば、API経由の取得自体は機械的。ただし小地域・地域メッシュのAPI仕様の詳細（レスポンスのタグ構造等）を実装前に開発ガイド・APIの使い方ページで別途確認する必要がある（今回は「利用ガイド」「API仕様（一般部分）」までの確認に留まり、メッシュ特有のレスポンス構造は未確認）。
- **C（医療POI）**: 低程度の工数。国土数値情報は都道府県別ZIP配布で形式が統一されており、GMLパーサまたはシェープファイルパーサ（既存のGISライブラリで対応可能なことが多い）で取り込める。福祉施設は自治体別利用条件の確認という非技術的な追加工数がある。

### 4.3 docs/12 feeds.yaml拡張への示唆

現行のfeeds.yamlは`source.type`が`gtfs-data-jp` / `odpt` / `direct-url`の3種類（GTFS/GTFS-Flex取得を前提としたスキーマ）である。A/B/Cはいずれも**非GTFSの静的地理データ**であり、そのまま既存スキーマに載せることはできない。以下の拡張が必要になる:

1. `source.type`に`ksj-prefecture-zip`（国土数値情報の都道府県別ZIP、都道府県コードをパラメータ化）、`hokonavi-ckan`（ほこナビCKANのデータセットID指定）、`estat-mesh-api`（e-Stat API、appId・統計表ID・メッシュレベル指定）等の新規typeを追加する
2. 現行スキーマは「フィード=時刻表データ」を前提にした`feed_start_date`/`feed_end_date`等のフィールドを持つ（docs/12参照）が、A/B/Cは「更新頻度が年単位〜不定期」という性質のため、GTFSと同じ鮮度監視ロジックを適用すべきか要検討（T-16の既存課題と同種の判断が必要）
3. ライセンス表記が3種類とも異なる（PDL1.0/CC BY 4.0/CC BY 4.0一部制限）ため、6章のクレジット生成ロジック（CreditManifest）に「ライセンス種別」の列挙を拡張する必要がある
4. B（メッシュ統計）は「シャードに埋め込む集計済みデータ」という性質のため、GTFSのようなfetch→normalize→shardのパイプラインとは別の「build-time enrichment」ステップとして位置づける方が実装が素直（fetchステージの対象に含めず、shard生成ステージの追加入力として扱う）
