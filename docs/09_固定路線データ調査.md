## 1. 調査方法の詳細

### 1.1 ODPTセンターCKANカタログ

`https://ckan.odpt.org/dataset?q=検索語`のフルテキスト検索と、`https://ckan.odpt.org/organization/`の組織一覧・組織別データセット一覧を併用した。

**判明した検索の癖（重要）**:
- CKANの検索インデックスは主にデータセットの**タイトル・説明文**を対象としており、路線番号（例: 「梅70」）や駅名（例: 「八高線」「箱根ケ崎」）のような詳細メタデータ・路線名の文字列では基本的にヒットしない。組織名（事業者名）や大分類のキーワード（「バス」「鉄道」等）での検索は機能する
- 「都営バス」という通称では0件だが、正式名称「東京都交通局」では19件ヒットする、といった表記揺れの影響が大きい。組織一覧ページ（`/organization/`、全128組織・7ページ）から正式名称を特定し、それで再検索するのが確実
- 組織一覧ページの「Nデータセット」表示は、その組織ページに実際に遷移して確認しないと信頼できない場合がある（JR東日本は一覧上「0データセット」と出ており、実際に組織ページ`/organization/jreast`を開いても「データセットが見つかりません」で一致した。今回はこのケースのみ確認したが、他組織についても表示件数を過信せず必要に応じて組織ページ本体を開くことを推奨する）

### 1.2 GTFSデータリポジトリ公開API

公式API仕様書（`http://docs.gtfs-data.jp/api.v2.html`、Redocly生成のOpenAPIドキュメント）を取得し、正確なパラメータ名を確認した上でクエリを実行した。詳細は6章にまとめる。

`gtfs-data.jp/search`のブラウザ向け検索ページ、および`developer.odpt.org`（ODPT開発者サイト）はいずれも「You need to enable JavaScript to run this app.」が返るのみで、`web_fetch`では内容を取得できなかった（事前情報どおり）。

### 3.2 好材料

- **瑞穂町コミュニティバスに「箱根ケ崎駅東口循環コース」が2026年3月改正で新設**され、GTFSデータリポジトリに公開されている（累計ダウンロード数32,924件と利用実績もある）。これはMVPの核心である「箱根ケ崎駅⇄町内デマンド交通」の固定路線側の受け皿として理想的なデータ

## 4. 既存ドキュメントとの記述齟齬（重要な指摘）

`03_データセット一覧.md`は開催概要ページ（https://challenge2026.odpt.org/ja/opendata.html）の記述を転記したもので、「鉄道: JR東日本のGTFS時刻表、GTFS-RT列車ロケーション・運行情報ほか多数」と記載されている。しかし本調査でCKANカタログを直接調べた結果、JR東日本（`jreast`組織）のデータセットは0件であり、関連するJR東日本アイステイションズ（`jre-is`組織）の2件もチャレンジ2026限定ライセンスの運行情報・GTFS-RTのみで、静的GTFS時刻表は見当たらなかった。

考えられる理由（いずれも未確認・推測）:
1. 開催概要ページの記述は「準備が整い次第随時追加」（`01_開催概要_募集要項.md`に既記載）の対象であり、現時点（2026-07-02）ではまだ一般公開カタログに反映されていない
2. エントリー後にアクセス可能な`members-portal.odpt.org`側に、CKAN非公開のJR東日本データが別途存在する
3. JR東日本のデータは「特別協力」としてのコンテスト向け提供であり、CKANの通常カタログ構造とは別の配布経路（開発者サイトのAPI経由のみ等）を取っている

**この齟齬はMVPのスケジュールに直結するリスクとして明記する**: 6章の開発ロードマップで「〜2026/8: チャレンジ限定Flexデータへのアクセス確保」とあるのと同様に、JR東日本の鉄道データについても「エントリー後に実際にアクセスできるか」を早期に確認する項目として追加すべき。

**フィード数の数え方に関する注記**: 上記はAPIレスポンスのレコード数（≒`organization_id + feed_id`の組み合わせ数に近い近似値）であり、厳密な「フィード」の定義（1系統1フィードか、1事業者1フィードか）によって解釈が変わる。同一の`feed_id`文字列（例: "communitybus"）を異なる組織が使用するケースが多数確認されており、単純な`feed_id`のみのユニーク集計は組織をまたいだ名寄せができず不正確になる。また、廃止済み・運行終了済みのフィードも`target_date=all`指定により件数に含まれている（例: 群馬県玉村町「たまりん」2024年10月廃止、富山県の名鉄バス路線2022年廃止など）ため、**「現在有効なフィード数」はこの数字より少ない**点に注意。

### 5.3 データ量の見立て

- 瑞穂町GTFS-Flex実データ（`07_瑞穂町Flexデータ検分.md`参照）は展開後17,867バイト（12ファイル）という極小規模。コミュニティバスGTFS（`communitybus`フィード）も同程度〜数倍程度の小規模ファイルと予想される（未確認、実データ未取得のため推定）
- 全国565件以上のGTFSフィードのうち、大半はコミュニティバス・地方バス事業者の小規模フィードと推測され、都営バス・西武バスのような大都市圏事業者の全系統フィードは相対的にファイルサイズが大きくなる（ファイル自体は未取得のため具体的なバイト数は不明、推定に留まる）
- 全国パイプラインを設計する場合、565件超のフィードを個別にポーリングする必要があり、GTFSデータリポジトリAPI側は`file_last_updated_at`を都道府県単位でまとめて取得できるため、差分検出（前回取得時の`file_last_updated_at`と比較）による効率化が可能

## 6. GTFSデータリポジトリAPIの仕様要点（全国取り込みパイプライン設計用）

### 6.1 エンドポイント一覧

公式ドキュメント: `http://docs.gtfs-data.jp/api.v2.html`（Redocly生成、OpenAPI 2.0ベータ公開版）

| エンドポイント | 用途 | 主なクエリパラメータ |
|---|---|---|
| `GET https://api.gtfs-data.jp/v2/files` | GTFSファイルの検索（フィード単位ではなく個別ファイル単位） | `extent`（bbox: minx,miny,maxx,maxy）、`pref`（都道府県コード、整数。例: `pref=13`）、`target_date`（有効日付、`YYYY-MM-DD`または`all`。省略時は現在有効なもののみ）、`org_name`（組織名の前方一致）、`org_id`（組織IDの完全一致） |
| `GET https://api.gtfs-data.jp/v2/feeds` | GTFSフィード一覧を取得（フィード単位のメタ情報、`real_time`のURL等を含む） | `pref`、`org_name`、`org_id` |
| `GET https://api.gtfs-data.jp/v2/organizations/{organization_id}/feeds/{feed_id}` | 特定フィードの詳細情報（過去/将来世代のファイル一覧、`validation_result`含む） | `max_prev`（デフォルト0）、`max_next` |
| `GET https://api.gtfs-data.jp/v2/organizations/{organization_id}/feeds/{feed_id}/files/feed.zip` | GTFS ZIP本体の取得 | `rid`（`prev_3`〜`current`〜`next_3`/`next`、デフォルト`current`）、`uid`（特定ファイルのUUID指定） |
| `GET .../files/stops.geojson` | バス停の可視化用GeoJSON | `rid`、`uid` |
| `GET .../files/routes.geojson` | 経路の可視化用GeoJSON | `rid`、`uid` |
| `GET .../files/tracking.geojson` | バスの動きの可視化用JSON | `rid`、`uid` |

**パラメータ名の注意（実機検証済み）**: `pref`パラメータは`pref_id`ではなく`pref`が正しい名称（公式サンプルでは`pref=23`の例が示されているが、これは県コード指定であり実際に`pref=13`で東京都、`pref=14`で神奈川県の絞り込みが機能することを確認した）。パラメータなしで`/v2/files`を呼ぶと、都道府県コード順とは異なる並び（内部的な登録順または地域コード順）で全国データが返り、大量になるためレスポンスがツール側のトークン上限で途中切断されるリスクが高い。**全国取り込みパイプラインでは`pref`パラメータで都道府県ごとに分割して取得するのが安全**。

**ページネーション**: 仕様書上、`/v2/files`・`/v2/feeds`ともにページネーション用パラメータ（`page`, `limit`, `offset`等）の記載がなく、レスポンスに`total`件数等のメタ情報も含まれない。県単位で絞り込んでも件数が多い場合（今回の調査で長野県・高知県が該当）、レスポンス全体を一度に受け切れない可能性がある。厳密な全件取得が必要な場合は、`extent`（bbox）パラメータを使って地域をさらに細分化する、または`org_id`を使って組織単位で個別に取得するなどの分割戦略が必要になる。

### 6.2 レスポンス形式

**実機で確認された仕様書との差異**: `file_rid`の値は仕様書サンプルでは`"current"`のような具体的な世代識別子だが、実際に取得したレスポンスでは`"not_implemented_yet"`という値が返っており、**世代管理機能（`rid`によるprev/next切替）は現時点で未実装の可能性がある**。パイプライン設計では`rid`パラメータに依存せず、`file_uid`を版識別に使う方が安全。

`/v2/feeds`のレスポンスは`real_time`オブジェクト（`trip_update_url`, `vehicle_position_url`, `alert_url`, `update_interval`）を含み、GTFS-RT提供の有無・更新間隔（秒）をここから判定できる設計になっている（ただし今回サンプルで見た実データの`real_time`オブジェクトは主要フィールドが空文字だったケースが多く、GTFS-RT自体を提供していないフィードが大半と見られる。瑞穂町コミュニティバスも`real_time.update_interval: 0`で、GTFS-RTなしの静的データのみ）。

`/v2/organizations/{org}/feeds/{feed}`の詳細レスポンスには`total_download_count`（累計ダウンロード数）、`gtfs_files[].memo`（改正内容の自由文メモ）、`gtfs_files[].update_info`（更新種別のフラグ: `update_timetable`, `update_routes`, `update_stops`, `update_fare`, `update_available_period`, `temporary_timetable`, `others`）が含まれ、**改正内容を構造化フラグで判別できる**（実際の値は今回全て`true`のケースしか確認できていないため、`false`ケースの挙動は未確認）。

### 6.3 利用規約・ライセンスの注意

- フィードごとに`feed_license_id`・`feed_license_url`が個別設定されており、**全フィード共通の単一ライセンスではない**。今回確認した範囲では CC BY 4.0（多数派）、CC0 1.0（例: 国立市くにっこ、大新東レイディアントシティ横濱）が混在していた
- ODPTセンター（CKAN）側は「公共交通オープンデータ基本ライセンス」（`odpt-ptodbl`、developer.odpt.org/termsで定義。166件で最多）とCC BY 4.0（90件）が主要ライセンスで、こちらも単一ではない。西武バスは基本ライセンス、都営バスはCC BY 4.0という違いが実際に確認された
- 「公共交通オープンデータ基本ライセンス」は、多くのデータセット説明文に「特定利用条件をよく読んで、ご利用ください」という注記が付されており、**CC BY 4.0よりも事業者ごとの追加条件が課される可能性がある**（本調査では利用規約本文（https://developer.odpt.org/terms ）自体は取得を試みていない。次のステップで内容確認が必要）
- 全国取り込みパイプラインでは、フィード単位でライセンスIDを保持し、表示・二次利用時にライセンスごとのクレジット表記を動的に切り替える設計が必須（CC BY系は「クレジット表示に関する情報」欄の提供者名をそのまま使えばよいが、基本ライセンス系は個別の特定利用条件を確認する必要がある）
- チャレンジ限定ライセンス（`odpt-ptodcll`）のデータは、コンテスト応募規約により「チャレンジ用データの目的外利用・第三者提供の禁止」「データ提供はチャレンジ終了までの予定」という制約がある（`01_開催概要_募集要項.md`既記載）。全国展開・コンテスト後の継続運用を前提とする場合、チャレンジ限定データに依存する設計は避けるべき

### 7.1 MVPの経路探索範囲の見直し

- **箱根ケ崎駅を「乗換ノード」として扱う場合、鉄道側（八高線）の時刻表データが現時点で確保できていない**ため、MVPの第一段階では「デマンド交通⇄箱根ケ崎駅」までの案内に留め、「駅から先の鉄道移動」は乗換案内アプリへのリンク誘導、または駅名表示のみの簡易対応にする代替案を用意すべき。エントリー後にJR東日本データへのアクセス状況を再確認し、確保できれば統合範囲を広げる
- **瑞穂町コミュニティバス（GTFSデータリポジトリ）の統合を優先度高く組み込む**。箱根ケ崎駅東口循環コースが2026年3月に新設されたばかりで、MVPの「デマンド交通と接続する固定路線」という要件に最も直接的に応える。既存の`07_瑞穂町Flexデータ検分.md`のパーサ実装知見（location_group展開等）とは別に、**通常の固定時刻GTFS**としての実装（従来型RAPTOR等）がこちらには必要になる点に注意（Flexの特殊パーサとは別レイヤ）
- 都営バス（梅70系統含む可能性）・西武バスは事業者単位のGTFSとして確保できているため、実データをダウンロードして`routes.txt`・`stops.txt`を確認し、箱根ケ崎駅・瑞穂町内停留所を実際にカバーする系統を特定する作業が次に必要（本調査はメタデータレベルに留まる）

### 7.2 データ取得パイプライン設計

- ODPT側（CKAN・developer.odpt.org経由）とGTFSデータリポジトリ側で、**取得方式を分けて実装する**必要がある。前者はODPT独自API（`api.odpt.org`/`api-public.odpt.org`）またはCKANリソースURL経由、後者は今回検証した`api.gtfs-data.jp/v2/`のREST APIで、認証不要かつパラメータでの絞り込みが可能
- GTFSデータリポジトリAPIは`pref`パラメータで都道府県別に分割取得するのが安全（全国一括取得はレスポンスサイズの問題が実際に発生した）。都道府県別に取得→`organization_id + feed_id`で名寄せ→差分検出（`file_last_updated_at`比較）という流れが妥当
- 瑞穂町の「チョイソコみずほまち」がODPT・GTFSリポジトリの両方に存在する事実を踏まえ、**同一自治体のデータが複数チャネルに重複公開されるケースがある**ことをパイプライン設計上の前提とし、重複統合ロジック（同一路線・同一事業者を検出して片方を正とする、または版が異なる場合は両方保持して差分を記録する）を組み込む
- 更新頻度の公式な保証が得られなかったため、**ポーリング頻度は保守的に（例: 日次）設定し、`file_last_updated_at`の変化検出で実際の更新タイミングを学習的に把握する**運用が現実的

### 7.3 ライセンス・クレジット表示の実装

- フィードごとの`feed_license_id`をデータモデルに保持し、UI上でクレジット表示を動的に生成する仕組みを早期に用意する（CC BY 4.0の「コンテンツ等の提供者名」欄の値をそのまま使う設計が、ODPT側では既に各データセットページに明記されているため転記しやすい）
- 「公共交通オープンデータ基本ライセンス」の特定利用条件は本調査では確認していないため、西武バス等このライセンスのデータを実際に使う前に`https://developer.odpt.org/terms`の内容確認を次のステップに追加する
