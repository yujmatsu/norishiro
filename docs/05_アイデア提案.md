# アイデア提案（2026-07-02版）

前提: 詳細は `01〜04` の各MDを参照。
狙いの定石 = **①今年の新データ（GTFS-Flex／地方都市圏／駅構内図／ほこナビ）を主役にする ②公式明示の社会課題（交通空白・少子高齢化・オーバーツーリズム・バリアフリー）に直結 ③過去9年の既出領域を外す**。

---

## ★推薦1: 「ノリシロ」— 日本初のデマンド交通対応ドアtoドア経路検索 ＋ AI移動コンシェルジュ

**一言**: 「その町、クルマがなくても行けます」。固定路線＋デマンド交通＋シェアサイクル＋フェリーを繋いだ、交通空白地帯を"消す"経路検索。

- **課題**: 地方では乗換案内アプリにデマンド交通（予約制乗合）が出てこないため、実際は移動手段があるのに「行けない」と判断される。予約方法も電話中心で心理障壁が高い。交通空白＝COMmmmONSの中心課題そのもの
- **中身**:
  - GTFS-Flex（エリア・予約ルール）を解釈する経路検索。世界的にも実装例が少なく（OTP2のFlexサポートを拡張）、「ここからここまではデマンド交通で移動可能。前日17時までに要予約、電話番号はこちら」まで案内
  - LLMコンシェルジュが対話で計画立案（「月曜の午前に◯◯病院へ行きたい」→ 予約締切逆算のリマインド付き行程）。高齢者向けに音声UI
  - 副産物として「デマンド込みの本当の到達圏マップ」を自治体向けに公開
- **データ**: ODPTデマンド交通GTFS-Flex（目玉）、GTFSリポジトリ（地方バス）、鉄道GTFS/GTFS-RT、GBFS、フェリーGTFS、e-Stat（病院・施設）
- **審査基準適合**: 社会課題◎（交通空白ど真ん中）／データインパクト◎（Flexの世界初級の消費事例→MobilityData審査員に刺さる、標準への還元も語れる）／技術◎（Flexルーティングは難度高いが誇れる）／UI/UX○（対話＋地図）
- **過去作品との差分**: Mobiviz（2025）はデマンド交通の"分析"。利用者向け経路検索への統合は皆無
- **リスク**: ルーティング実装の重さ → 対象地域を協力9自治体（白馬村・富岡市等）に絞ってMVP

## ★推薦2: 「ヒカゲジュンロ（日陰巡路）」— 気候適応ナビ：暑さ・雨をかわして移動する

**一言**: PLATEAUの3D都市モデルで日陰を計算し、「熱中症リスク最小」の歩行＋公共交通ルートを提案。

- **課題**: 猛暑の夏、高齢者・子連れ・観光客の熱中症搬送が急増。既存ナビは「最短」だけで「炎天下のバス停で12分待ち」を平気で提案する
- **中身**:
  - PLATEAU建物モデル＋太陽位置から時刻別の影マップを生成し、日向曝露が最小の徒歩経路を計算。地下通路・屋根付き通路（駅構内図・ほこナビ）も編入
  - GTFS/GTFS-RTでバス停ごとの「待ち時間×日射」を算出し、「1本遅らせて屋内で待つ」「日陰側のバス停から乗る」を提案
  - 気象庁データ（気温・日射・WBGT・雨雲）で夏は暑熱、雨天は濡れない経路に自動切替。冬は逆に「日なたルート」
  - 3D日陰マップの見た目が強く、デモ映えする
- **データ**: PLATEAU（強推奨データ）、ほこナビ歩行空間NW（強推奨）、駅構内図（今年充実）、GTFS/GTFS-RT、GBFS、気象庁
- **審査基準適合**: 社会課題◎（気候適応×高齢者、独自性ある切り口）／データインパクト◎（PLATEAU×ほこナビ×気象×交通の4種マッシュアップ）／技術◎（影計算＋ルーティング）／UI/UX◎（3D可視化の説得力）
- **過去作品との差分**: 気象は「めぐる」が遅延予測に使ったのみ。日陰・暑熱ルーティングは皆無。SPECTRAはPLATEAUを"背景"に使ったが、本案は3Dモデルを"計算に"使う
- **リスク**: 影計算の負荷 → 事前計算タイル化で解決可能

## 案3: 「卒くる」— 運転免許返納シミュレーター

- **一言**: 「お父さん、クルマなしでも暮らせるか、データで一緒に見てみよう」
- 自宅と週間の移動パターン（通院・買い物・趣味）を入れると、全便をGTFS＋GTFS-Flexで代替経路に変換し、**クルマなし生活の実現度スコア**、月間コスト比較（車維持費 vs 交通費）、歩行量の健康効果を提示。家族と共有できる「返納検討レポート」を出力
- データ: GTFS/Flex、e-Stat、警察庁事故統計（高齢運転リスクの提示）、医療機関データ
- 差分: 住みログ/relocateは「引越し先選び」。"今の家で車を手放せるか"は未開拓。高齢ドライバー事故という時事性の強い社会課題
- リスク: 経路検索基盤が必要（案1と部品共通化可能）

## 案4: 「みんなの駅コンパス」— 感覚・認知特性対応の駅構内ナビ

- **一言**: 車椅子の"次"のバリアフリー。発達障害・認知症・パニック障害の人が駅を使えるように
- 駅構内図＋ほこナビで、混雑・騒音・光刺激を避ける「センサリーマップ」、易しい言葉のターンバイターン案内、不安時の「静かな待避場所」表示。エレベーター停止情報のクラウドソーシング
- データ: 駅構内図（今年充実）、ほこナビ、GTFS-RT、混雑データ
- 差分: EkiLink/わたしの構内図は車椅子・段差対応。**感覚過敏・認知面**のアクセシビリティは9年間ゼロ。別所審査員（障がい者支援・BLE混雑）と親和性大
- リスク: 感覚データの整備を自前で行う必要（音量実測など実地調査が武器にもなる）

## 案5: 「Transit MCP」— 公共交通オープンデータのAIエージェント基盤

- **一言**: 日本の公共交通データを世界中のLLMエージェントから使えるようにするインフラ
- ODPT API・GTFSリポジトリ・GTFS-RT・GBFSをMCPサーバー化し、Claude/ChatGPT等が経路計画・遅延監視・データ分析をツールとして実行可能に。デモとして多言語旅行コンシェルジュと自治体データ分析チャットを同梱
- データ: 全部（横断基盤なので）
- 差分: TransitHQ（2018）のAPI提供構想のAI時代版。エージェント基盤は皆無。「オープンデータ活用のインパクト」基準に最も直接的に応え、Postmanエバンジェリスト・坂村会長（オープン基盤思想）に刺さる
- リスク: エンドユーザーUIが弱くなりがち → デモアプリの完成度で補う。案1のコンシェルジュ層としての合体も可

## 案6: 「ずらしステイ&ゴー」— オーバーツーリズム負荷分散エンジン

- **一言**: 混雑ピークを避けると得をする観光ナビ
- リアルタイム運行・混雑推定＋観光POIで観光地の「今の混み」を予測し、LLMが「雰囲気が似た空いている代替スポット」と、そこへ向かう空いた時間帯の公共交通を提案。多言語対応
- データ: GTFS-RT、観光オープンデータ、人流（G空間）、e-Stat
- 差分: まちマッチング（ランダム分散）、エモルート（構想止まり）を「混雑データ駆動」で実装まで持ち込む点が新規。ただし方向性としては既出に近く、混雑データの入手性が鍵
- リスク: 混雑の真値データが乏しい

## 案7: 「のせバス」— 貨客混載マッチング

- **一言**: 空気を運んでいる地方バスの座席・トランクを、買い物代行と小口配送に使う
- 商店・薬局の配達依頼と、GTFS時刻表上の便をマッチングし、バス停で受け渡し。2024年問題×交通空白×買い物難民を一度に撃つ
- データ: GTFSリポジトリ、デマンド交通、e-Stat（商業統計）
- 差分: OcRail（貨物可視化）とは別物。完全未開拓
- リスク: 実運用には事業者合意が必要で、コンテスト期間内は「シミュレーション＋予約モック」に留まる可能性。審査の「技術的完成度」で不利になり得る
