## 1. スコープと非スコープ

### 1.1 v1でやること

- 出発時刻指定でのマルチクライテリア（到着時刻×乗換回数）Pareto最適経路探索。固定路線（鉄道・バス）とGTFS-Flex（location_group型）の統合探索。
- 徒歩アクセス・イグレス（起点/終点が停留所そのものでない場合の補完）、停留所間の徒歩乗換。
- Flexレッグの時間窓判定・予約締切判定（`booking_type∈{0,1,2}`）を経路の実行可能性フィルタとして適用。
- 単一サービス日内の探索（日付をまたぐ深夜運行は「サービス日の24時超え秒表現」で単一シャード内に収める。10_GTFS-Flex実装仕様.md 3.4節が想定する範囲と同じ）。
- 到達圏（isochrone）算出。自治体向け副産物・審査デモ用。
- 都道府県シャード規模（stop数・trip数が瑞穂町の数十〜百倍程度のオーダー）を想定したデータ構造・性能設計。

### 1.2 v1で扱わないもの（非スコープ）

以下はいずれも将来拡張の対象であり、v1のAPI・データ構造の変更を最小化する形で拡張余地を残すが、v1では実装しない。

### 1.3 v1の探索空間の境界

- 探索は単一の「結合済みグラフ」（出発地・目的地が属する地域シャード＋鉄道バックボーンシャードをWorker内で結合したもの、`11_アーキテクチャ設計.md` 3章(a)）に対して行う。シャード境界をまたぐ動的な追加ロードはRAPTOR実行中には発生しない（実行前に必要なシャードを揃えてから`plan()`を呼ぶ、9.4節）。
- `maxTransfers`はRAPTORのラウンド数上限と同義（3.3節）。指定がない場合のデフォルト値は実装定数として設ける（例: 6）。
- `walkLimit`は徒歩アクセス/イグレス/乗換の距離上限（デフォルトは確定済み設計判断3の800m、5章）。

## 2. データモデル

### 2.1 設計方針

`packages/router`は`packages/gtfs`のNormalized型を直接読まない（`11_アーキテクチャ設計.md` 2章の依存方向ルール）。`apps/pipeline`が事前にNormalized型から`12_データパイプライン設計.md` 4章が規定するシャード形式（**v1では圧縮JSON、列指向・配列インデックス参照方式**。同書の確定済み設計判断4「v1は圧縮JSON、性能不足が判明した時点でバイナリ化を検討する」）へ変換し、`packages/router`はそのJSONシャードを読み込む。

**本章が定義する`RouterShard`はシャードファイルそのものの形式ではなく、`loadShard()`（9.4節）がJSONシャードを読み込んだ直後に構築する「ランタイム内部表現」である。** シャードファイル自体（`12_データパイプライン設計.md` 4.2〜4.5節の`Shard`/`CompressedStops`/`CompressedStopTimes`/`FlexData`/`CompressedTransfers`型、`number[]`中心のJSON配列）と本章の`RouterShard`（TypedArray中心）は別レイヤであり、両者の対応関係は9.2節で明示する。この2層分離により、シャードのファイル形式（JSON、デバッグ時に目視確認しやすいことを`12_データパイプライン設計.md`が優先した理由）とルーティングコアのホットパス最適化（TypedArrayによるメモリ局所性、9.1節）を、互いの確定済み設計判断を変更せずに両立させる。

RAPTORは「ラウンドごとに、あるtripに乗っている区間で到達可能な停留所を更新する」処理を繰り返す。この処理を高速化する標準的なデータ構造がCSR（Compressed Sparse Row）的な配列表現である。route単位でtripをまとめ、trip単位でstop_timesを平坦化した1次元配列に格納することで、「あるrouteの何番目のtripの何番目の停留所か」をポインタ演算だけで求められる。`12_データパイプライン設計.md`の`CompressedStopTimes`（`tripIdx`/`stopIdx`のインデックス参照配列、4.3節）は既にrow-oriented ID再掲を避けた形式であり、本章のCSR変換（`loadShard()`内、9.4節）はこれをルート単位のオフセット配列に再構成する処理として位置づく（ゼロからのパースではなく、既に配列化されたJSONの並べ替えで済む）。

### 2.2 コア構造

```typescript
// packages/types で定義し、packages/router が読み込む「シャードのランタイム表現」

/** 停留所インデックス。シャード内でstop_idを0始まりの連番に正規化したもの */
type StopIdx = number;
/** ルートインデックス */
type RouteIdx = number;
/** トリップインデックス（シャード内グローバル連番） */
type TripIdx = number;
/** サービス日の0時からの秒。24:00を超える表現を許す（確定済み設計判断2） */
type SecondsSinceServiceDayStart = number;

interface RouterShard {
  // --- 停留所 ---
  /** StopIdx -> stop_id（元のGTFS ID）の逆引き。表示・レッグ復元用 */
  stopIds: string[];
  /** StopIdx -> 緯度。Float64Array。Haversine計算・isochrone・point-in-polygon(将来)用 */
  stopLat: Float64Array;
  /** StopIdx -> 経度 */
  stopLon: Float64Array;

  // --- ルートとトリップ（CSR的配列） ---
  /**
   * RouteIdx -> そのrouteに属するTripIdxの開始位置。
   * routeTripsStart[r] .. routeTripsStart[r+1]-1 が route r のトリップ範囲（CSR標準のoffset配列）。
   * 長さは routeCount + 1（末尾に総トリップ数を置く番兵）。
   */
  routeTripsStart: Uint32Array;
  /** CSR本体。routeTripsStartの範囲でスライスして得られるTripIdxの列。route内で最も早い出発時刻の順にソート済み（3.2節のスキャンで単調性を利用するため） */
  routeTrips: Uint32Array;

  /**
   * RouteIdx -> そのrouteが経由するStopIdxの並び（stop_sequence順）の開始位置。
   * 同一route内の全tripは同じ停留所列を共有する前提（GTFSの標準的な仮定。stop_times CSR配列と対応するインデックス計算の基盤）。
   * 長さは routeCount + 1。
   */
  routeStopsStart: Uint32Array;
  /** CSR本体。route内の停留所列（StopIdxの並び）。 */
  routeStops: Uint32Array;

  /**
   * StopTimes本体。TripIdx×route内停留所インデックスの2次元アクセスを1次元に平坦化。
   * あるトリップtの、route内i番目の停留所の出発時刻は
   *   stopTimesDeparture[tripStopTimesOffset[t] + i]
   * で求める（tripStopTimesOffset[t] は routeStopsStart[route(t)] と同じ長さのオフセットを共有できるよう、
   * トリップは必ず「自分が属するrouteの停留所列と同じ長さ」を持つことをビルド時に保証する。9.2節）。
   */
  stopTimesArrival: Int32Array;
  stopTimesDeparture: Int32Array;
  /** TripIdx -> どのRouteIdxに属するか */
  tripRoute: Uint32Array;
  /** TripIdx -> サービス日に運行するか判定するためのビットセット参照（2.3節） */
  tripServiceIdx: Uint32Array;

  // --- 停留所→通過routeの逆引き（ラウンド処理でのスキャン起点） ---
  /** StopIdx -> そのstopを経由するRouteIdxの列の開始位置。長さ stopCount + 1 */
  stopRoutesStart: Uint32Array;
  /** CSR本体 */
  stopRoutes: Uint32Array;

  // --- 徒歩transfer（事前計算済み、確定済み設計判断3） ---
  /** StopIdx -> 徒歩到達可能な近傍StopIdxの列の開始位置。長さ stopCount + 1 */
  transfersStart: Uint32Array;
  /** CSR本体。到達可能な近傍StopIdx */
  transfersTo: Uint32Array;
  /** transfersToと同じ長さ。徒歩所要秒（5章の式で事前計算済み） */
  transfersDurationSec: Int32Array;

  // --- サービス日判定 ---
  /** シャードが表現する単一の対象サービス日（YYYYMMDD）。シャードは日付別に展開済み（確定済み設計判断2） */
  serviceDate: number;
  /** ServiceIdx -> その日に運行するかのビットセット。1トリップ=1ビット。tripServiceIdxが指すのはこの配列のビット位置 */
  activeServiceBits: Uint8Array;

  // --- Flex ---
  flexGroups: FlexGroupTable;

  // --- 逆引き（レッグ復元用、6章） ---
  routeIds: string[];
  tripIds: string[];
}
```

**設計上の要点**:

- `routeStops`と`stopTimesArrival`/`stopTimesDeparture`の添字対応: 標準的なRAPTOR実装（Delling et al. 2015）に合わせて、「route内で共通の停留所列」という仮定を置く。GTFSでは同一route_idの複数tripが停留所リストの異なるバリエーション（一部区間運休等）を持つことがあるため、`apps/pipeline`のビルド時に「実質的に異なる停留所列を持つtripは別のRouteIdxとして分割する」正規化を行う（**同一route_idが複数のRouteIdxに分割されることを許容する**、9.2節）。これにより`packages/router`側は「1 RouteIdx = 1停留所列パターン」という単純な前提だけで実装できる。
- `Int32Array`を使う時刻表現は秒単位（`SecondsSinceServiceDayStart`）。24時間=86400秒を超える値（深夜運行の24:00超え表現）も`Int32Array`の範囲（約±21億）に十分収まる。
- 全てTypedArrayで表現するのは9章の性能設計（メモリ局所性、GCプレッシャの低減）と直結する。

### 2.3 カレンダー・サービス日展開

確定済み設計判断2「calendar/calendar_datesはパイプラインで日付別に展開済み（シャード内はサービス日→trip集合）」を反映し、`packages/router`はcalendar.txt/calendar_dates.txtの曜日ロジック・例外日ロジックを一切持たない。`apps/pipeline`が対象サービス日ごとに「その日運行するtripのビットセット」を計算し、シャードに`activeServiceBits`として埋め込む。`tripServiceIdx[t]`はそのビット位置を指す1つのインデックスであり、判定は`isBitSet(activeServiceBits, tripServiceIdx[t])`というO(1)ルックアップになる（3章のスキャンで毎回行われる高頻度操作のため、O(1)が必須）。

このシャードは**単一サービス日専用**である（`serviceDate`フィールド）。別の日付を探索する場合は別シャードをロードする（1.3節、NS-4）。

### 2.4 停留所インデックスとID解決

`stopIds: string[]`は`StopIdx -> stop_id`の逆引き用。順方向（`stop_id -> StopIdx`）の解決は、シャードロード時に一度だけ`Map<string, StopIdx>`を構築してキャッシュする（`RouterShard`自体には含めず、シャードをラップする`ShardIndex`のような実行時オブジェクトが持つ。頻繁な文字列比較をホットパスから排除するため、ラウンド処理内部は常に`StopIdx`の数値だけで動作し、文字列IDへの変換は結果復元（6章）の最後の1回だけで行う）。

座標→最寄り停留所解決（`PlanRequest`が座標で起点/終点を指定するケース、8章）は、シャードロード時に構築する空間インデックス（グリッドまたはKD-tree、9.3節）を介して行う。

### 2.5 Flexグループのデータ構造

`10_GTFS-Flex実装仕様.md` 3.1節の`FlexLeg`概念と3.3節の乗車専用行/降車専用行判定を、シャードのランタイム表現として構造化する。

```typescript
/** FlexGroupIdx: シャード内でlocation_group_idを0始まりの連番に正規化したもの */
type FlexGroupIdx = number;
/** FlexTripIdx: Flexサービスの「1トリップ」（例: east_trip）を表す連番。固定路線のTripIdxとは別の名前空間 */
type FlexTripIdx = number;

interface FlexGroupTable {
  /** FlexGroupIdx -> 元のlocation_group_id文字列 */
  flexGroupIds: string[];

  /**
   * FlexGroupIdx -> 所属StopIdxの列の開始位置（CSR）。
   * 10_GTFS-Flex実装仕様.md 3.1節ステップ1「Map<location_group_id, Set<stop_id>>」のCSR表現。
   * 長さ flexGroupCount + 1。
   */
  groupStopsStart: Uint32Array;
  /** CSR本体。グループに属するStopIdxの列（重複なし） */
  groupStops: Uint32Array;

  /**
   * StopIdx -> そのstopが属するFlexGroupIdxの列の開始位置（逆引きCSR）。
   * 4章のラウンド処理は「marked stopがどのFlexGroupに属するか」を高頻度でルックアップするため、
   * 順方向（group->stops）だけでなく逆方向（stop->groups）のCSRも保持する。
   * 長さ stopCount + 1。
   */
  stopGroupsStart: Uint32Array;
  /** CSR本体 */
  stopGroups: Uint32Array;

  /**
   * FlexTripIdx -> どのFlexGroupIdxを対象にするか。
   * 10_GTFS-Flex実装仕様.md 3.3節「1トリップにpickup_only1行・dropoff_only1行」を一般化し、
   * 1 FlexTripIdx が複数のpickup_only/dropoff_only行を持てるようにする（3.3節の汎用化注意点に対応）。
   */
  flexTripGroup: Uint32Array;
  /** FlexTripIdx -> サービス日判定に使うビット位置（2.3節のactiveServiceBitsと同じ配列を共有） */
  flexTripServiceIdx: Uint32Array;

  /**
   * pickup_only行の集合。1 FlexTripIdx に対して複数行がありうるため、CSRで表現。
   * pickupRowsStart[ft]..pickupRowsStart[ft+1]-1 が FlexTripIdx=ft の pickup_only行群。
   */
  pickupRowsStart: Uint32Array;
  pickupWindowStart: Int32Array;   // 各pickup_only行のstart_pickup_drop_off_window（秒）
  pickupWindowEnd: Int32Array;     // 各行のend_pickup_drop_off_window（秒）

## 3. RAPTOR本体の擬似コード

### 3.1 用語・記号

標準的なRAPTOR論文（Delling, Pajor, Werneck, "Round-Based Public Transit Routing", 2015）の記法に合わせる。

- `k`: ラウンド番号（0, 1, 2, ...）。ラウンド`k`終了時点の`τ_k[p]`は「乗換`k`回以内でstop `p`に到達できる最早到着時刻」を意味する。
- `τ[p]`: stop `p`への現在の最良到着時刻（ラウンド間で共有・更新される、最終的な最早到着時刻の下限）。
- `τ_k[p]`: ラウンド`k`終了時点の到着時刻（マルチクライテリアのPareto集合を保持するため、7章のisochrone・6章の再構成で使う）。
- `markedStops`: 直前のラウンドで到着時刻が改善されたstopの集合。次ラウンドのスキャン起点。
- `Q`: ラウンド内で「スキャンすべきroute」を集める作業集合（各routeについて「そのroute内で最も早いstop_sequence」だけを覚える、標準RAPTORの`Q`に相当）。

### 3.2 初期化

```
function initialize(shard: RouterShard, source: AccessLegSet, departureTime: number):
    // τ, τ_k を全stopについて+∞で初期化
    τ ← array of size shard.stopCount, filled with +Infinity
    τ_prev ← copy of τ            // ラウンドk-1終了時点のスナップショット（3.3節のローカルプルーニングに使用）
    bestArrival ← array of size shard.stopCount, filled with +Infinity  // 全ラウンド共通の最良値（3.3節のグローバル枝刈り用）

    markedStops ← empty set
    journeyLegs ← empty map<StopIdx, LegRecord>  // 6章の復元用。stopごとに「そのstopへの最良到達を実現した直前のレッグ」を記録

    // source: 出発地からの徒歩access（5.1節）。座標指定ならaccessレッグ経由、stopId指定ならτ[stop]=departureTimeの1点
    for (stopIdx, walkDurationSec) in source.reachableStops:
        arrival ← departureTime + walkDurationSec
        if arrival < τ[stopIdx]:
            τ[stopIdx] ← arrival
            bestArrival[stopIdx] ← arrival
            markedStops.add(stopIdx)
            journeyLegs[stopIdx] ← { kind: "walk", from: source, to: stopIdx, durationSec: walkDurationSec }

    return { τ, τ_prev, bestArrival, markedStops, journeyLegs }
```

`AccessLegSet`はorigin（座標 or stopId）から徒歩到達可能なstopの集合（5.1節で計算方法を定義）。origin自体がstopIdの場合は`reachableStops = [(stopIdx, 0)]`の1要素になる。

### 3.3 ラウンド処理（本体）

RAPTORの1ラウンドは「(a) 固定路線tripのスキャンによる改善」「(b) Flex仮想レッグのスキャンによる改善（4章）」「(c) 徒歩transferによる緩和（5.3節）」の3段階からなる。標準RAPTORでは(a)のみだが、本エンジンは(b)を確定済み設計判断4に従い同じラウンド内の拡張ステップとして追加する。

```
function run(shard: RouterShard, req: PlanRequest): Itinerary[]:
    state ← initialize(shard, resolveAccess(req.origin, shard), req.departureTime)
    maxK ← req.maxTransfers ?? DEFAULT_MAX_TRANSFERS   // 既定値 6

    for k in 1..maxK:
        if state.markedStops.isEmpty(): break   // 打ち切り: これ以上改善する見込みがない

        roundMarked ← empty set   // このラウンドで新たに改善されたstop（次ラウンドの起点）

        // --- (a) 固定路線tripのスキャン ---
        Q ← collectRoutesServing(shard, state.markedStops)
        // Q: RouteIdx -> そのroute内で「乗車を検討し始めるべき最小のstop_sequence位置」
        //    標準RAPTORの「各routeにつき最も早いmarked stopの位置から走査すればよい」最適化

        state.τ_prev ← copy of state.τ   // このラウンド開始時点のスナップショット
        state.markedStops.clear()

        for (routeIdx, boardFromSeqIdx) in Q:
            scanRoute(shard, routeIdx, boardFromSeqIdx, state, k, req.targetPrune, roundMarked)

        // --- (b) Flexレッグのスキャン（4章で詳述、このラウンドで新たにmarkされたstopに対して行う） ---
        scanFlexLegs(shard, roundMarked, state, k, req, roundMarked)

        // --- (c) 徒歩transferの緩和（5.3節） ---
        relaxTransfers(shard, roundMarked, state, k)

        state.markedStops ← roundMarked

    return reconstructItineraries(shard, state, req)  // 6章
```

**`scanRoute`（標準RAPTORの核。1つのrouteについて、乗車位置から先頭方向に停留所を走査し、乗っているtripを必要なら乗り換えながら到達時刻を更新する）**:

```
function scanRoute(shard, routeIdx, boardFromSeqIdx, state, k, targetPrune, roundMarked):
    stopsInRoute ← shard.routeStops[shard.routeStopsStart[routeIdx] .. shard.routeStopsStart[routeIdx+1]-1]
    currentTrip ← NONE   // 現在「仮に乗っている」と仮定しているtrip（まだ確定的に乗車したわけではない、標準RAPTORの遅延評価）
    boardedAtSeqIdx ← -1

    for seqIdx in boardFromSeqIdx .. stopsInRoute.length - 1:
        stopIdx ← stopsInRoute[seqIdx]

        // (i) 現在乗っているとみなしているtripがあれば、この停留所での到着時刻を見て改善判定
        if currentTrip != NONE:
            arrivalSec ← stopTimeArrival(shard, currentTrip, seqIdx)
            if arrivalSec < min(state.τ[stopIdx], targetPrune):   // targetPrune: 目的地の現在最良到着時刻（枝刈り、3.4節）
                state.τ[stopIdx] ← arrivalSec
                state.bestArrival[stopIdx] ← min(state.bestArrival[stopIdx], arrivalSec)
                roundMarked.add(stopIdx)
                state.journeyLegs[stopIdx] ← {
                    kind: "transit", tripIdx: currentTrip, routeIdx,
                    boardStopIdx: stopsInRoute[boardedAtSeqIdx], alightStopIdx: stopIdx,
                    boardSeqIdx: boardedAtSeqIdx, alightSeqIdx: seqIdx, round: k,
                }

        // (ii) この停留所で、より早く出発できるtripに乗り換えられるか判定
        //      「ラウンドk-1終了時点でこのstopに到達していた時刻」から、乗車可能な最速tripを探す
        //      (標準RAPTORの単調性: routeTripsが出発時刻順にソート済みなので、
        //       前方に進むほど「乗るべきtrip」は同じか後ろにしか進まない -> 逐次的に前進するだけでO(1)償却)
        if state.τ_prev[stopIdx] < currentDepartureBound(shard, currentTrip, seqIdx):
            candidateTrip ← earliestTripDepartingAtOrAfter(shard, routeIdx, seqIdx, state.τ_prev[stopIdx])
            if candidateTrip != NONE and (currentTrip == NONE or candidateTrip is earlier-or-equal than currentTrip):
                currentTrip ← candidateTrip
                boardedAtSeqIdx ← seqIdx
```

**計算量**:

- `collectRoutesServing`: markedStopsの各stopについて`stopRoutes`のCSRを走査するので `O(Σ deg(stop))`。標準RAPTORの解析と同じ。
- `scanRoute`全体（1ラウンド）: 各routeを高々1回、route内の停留所列を1回だけ前進で走査するため、ラウンドあたり`O(sum of route lengths touched)`。標準RAPTORの理論計算量`O(rounds × (routes + transfers))`に一致する（Delling et al. 2015の解析）。
- `earliestTripDepartingAtOrAfter`は、標準RAPTORでは「同じrouteを前方に進むにつれて探索対象のtrip添字も単調に前進する」性質を使い、routeごとに1回の線形走査で償却O(1)にできる（trip配列が出発時刻順にソート済み、2.2節の`routeTrips`の定義）。実装では二分探索でも正当だが、償却定数のためには単調ポインタ方式を推奨する。
- 全体の計算量: ラウンド数を`K`（`= maxTransfers`）、固定路線部分の走査総量を`R`とすると `O(K × R)`。これに4章のFlexスキャンの計算量が加わる（4.5節）。

### 3.4 目的地枝刈り（target pruning）

標準RAPTOR最適化として、目的地（`req.destination`が指す1つ以上のStopIdx、egressの逆写像）の現在の最良到着時刻を`targetPrune`として保持し、`scanRoute`内の改善判定に使う（`arrivalSec < min(state.τ[stopIdx], targetPrune)`）。目的地までの現在の最良到達より遅い改善は最終的にPareto最適解に寄与しないため早期に捨てる。この枝刈りは6.1節のPareto集合構築と整合するよう、「乗換回数が少ないラウンドでの目的地到着」も別途保持する（6.1節）。

## 4. Flex仮想レッグ注入の擬似コード

### 4.1 注入のタイミング

確定済み設計判断4「Flexトリップは仮想レッグとしてRAPTORの各ラウンドの拡張ステップで注入する」に従い、3.3節の`run()`本体が示す通り、**各ラウンドの固定路線スキャン（a）が完了した直後、かつ徒歩transfer緩和（c）より前**に実行する。理由:

- 固定路線スキャンで新たに到達可能になった停留所（`roundMarked`）が、同じラウンド内でFlexグループへの乗車起点になり得る（例: バスで来た停留所からデマンド交通に乗り継ぐ）ことを1ラウンドの中で捉える。
- 徒歩transferより先にFlexを評価するのは、Flexレッグ自体が「乗換1回」を消費する乗り物的な移動（10_GTFS-Flex実装仕様.md 3.5節「Flexレッグも1回の乗換としてラウンド番号を1つ消費させるのが単純で安全」）であるのに対し、徒歩transferは同一ラウンド内の「無料の乗換」（5.3節）として扱うためであり、順序を入れ替えるとFlexレッグ経由で到達した停留所からさらに徒歩transferで別の停留所に緩和する、という自然な依存関係になる。

### 4.2 対象とする停留所集合

`10_GTFS-Flex実装仕様.md` 3.5節の疑似コードが示す通り、走査対象は**このラウンドで新たに改善されたstop（`roundMarked`）のみ**である。ラウンド開始前から既にPareto最適な到達時刻を持っていたstopを毎ラウンド再走査する必要はない（改善がなければFlexレッグ経由の到達時刻も変わらないため）。

```
function scanFlexLegs(shard, roundMarked, state, k, req, outMarked):
    for stopIdx in roundMarked:
        groupIndices ← flexGroupsContaining(shard.flexGroups, stopIdx)   // stopGroups CSRの逆引き

        for groupIdx in groupIndices:
            for flexTripIdx in flexTripsForGroup(shard.flexGroups, groupIdx):
                if not isFlexServiceActive(shard, flexTripIdx, req.serviceDate): continue

                // pickup_only行の集合を全て試す（2.5節: 1 FlexTripに複数のpickup_only行がありうる）
                for pickupRow in pickupRowsOf(shard.flexGroups, flexTripIdx):
                    tArrival ← state.τ[stopIdx]   // このstopへの現在の到着時刻 = Flexサービス側からの「乗車希望時刻」

                    // 時間窓判定その1: 乗車が窓内か
                    if not (pickupRow.windowStart <= tArrival <= pickupRow.windowEnd): continue

                    // 乗車可否: pickup_type=1(不可)の行はそもそもpickupRowsに含まれない（2.5節のテーブル構築時に除外済み）

                    for dropoffRow in dropoffRowsOf(shard.flexGroups, flexTripIdx):
                        // 中間の時間窓レコード無視ルール（10_GTFS-Flex実装仕様.md 2.6節）:
                        // pickupRowとdropoffRowの間に他のstop_sequenceが存在しても、
                        // この2点間を直接1レッグとして評価する（中間行を経由候補にしない）。
                        // groupStops自体が既に「グループに属する全stop」を平坦に持つため、
                        // 本ループは中間行の概念を最初から持たない実装になっている。

                        for otherStopIdx in stopsOfGroup(shard.flexGroups, groupIdx):
                            if otherStopIdx == stopIdx: continue
                            // 同一グループ内でも pickupRow の乗車対象と dropoffRow の降車対象が別の行に紐づく
                            // 場合があるため（2.5節のクロス積）、双方が同一グループを指していれば組み合わせ可能とみなす。

                            durationSec ← estimateFlexDuration(shard.flexGroups.durationEstimatorParams, shard, stopIdx, otherStopIdx)
                            tDropoff ← tArrival + durationSec

                            // 時間窓判定その2: 降車推定時刻も窓内に収まること（確定済み設計判断4）
                            if not (dropoffRow.windowStart <= tDropoff <= dropoffRow.windowEnd): continue

                            // 予約制約判定（確定済み設計判断4、4.4節で詳述）
                            bookingResult ← evaluateBooking(shard.flexGroups.bookingRules, pickupRow.bookingRuleIdx, tArrival, req.searchTime, req.serviceDate)
                            if not bookingResult.feasible: continue  // 実行不可能なレッグは採用しない（不可視ではなく、6.4節で参考情報として別保持）

                            if tDropoff < state.τ[otherStopIdx]:
                                state.τ[otherStopIdx] ← tDropoff
                                state.bestArrival[otherStopIdx] ← min(state.bestArrival[otherStopIdx], tDropoff)
                                outMarked.add(otherStopIdx)
                                state.journeyLegs[otherStopIdx] ← {
                                    kind: "flex",
                                    flexTripIdx, groupIdx,
                                    fromStopIdx: stopIdx, toStopIdx: otherStopIdx,
                                    departSec: tArrival, arriveSec: tDropoff,
                                    pickupBookingRuleIdx: pickupRow.bookingRuleIdx,
                                    dropoffBookingRuleIdx: dropoffRow.bookingRuleIdx,
                                    bookingDeadlineSec: bookingResult.deadlineSec,   // 6.4節でレッグに添付
                                    round: k,
                                }
```

### 4.3 所要時間推定関数

確定済み設計判断4の式をそのまま実装する（`10_GTFS-Flex実装仕様.md` 3.2節の選択肢Aと一致）。

```
function estimateFlexDuration(params: DurationEstimatorParams, shard, fromStopIdx, toStopIdx) -> number:
    distanceMeters ← haversine(
        shard.stopLat[fromStopIdx], shard.stopLon[fromStopIdx],
        shard.stopLat[toStopIdx], shard.stopLon[toStopIdx],
    )
    drivingDurationSec ← (distanceMeters * params.detourFactor) / params.averageSpeedMps
    return drivingDurationSec + params.boardingBufferSec
```

`params.detourFactor = 1.4`、`params.averageSpeedMps = 22*1000/3600 ≒ 6.111`、`params.boardingBufferSec = 300`（乗降バッファ5分）が既定値（確定済み設計判断4、2.5節の`durationEstimatorParams`）。`haversine`は5章で定義する徒歩用のHaversine関数と同一の実装を共有する（迂回係数・速度パラメータのみ異なる、9.1節でモジュール共有を明記）。

将来`mean_duration_factor`/`safe_duration_factor`が入力データに存在する場合の切替は、`10_GTFS-Flex実装仕様.md` 3.2節が示すハイブリッド方式（Haversine推定値を`DrivingDuration`とみなして係数を掛け合わせる）に従うが、v1のシャード構造（2.5節）にはこの切替のためのフィールドを持たせず、`apps/pipeline`のビルド時に係数が存在すれば`durationEstimatorParams`自体を事前に調整済みの値として書き込む方式にする（`packages/router`ランタイムは常に単純な線形式だけを評価すればよく、分岐を持たない）。

### 4.4 予約制約の評価

`10_GTFS-Flex実装仕様.md` 3.4節の`isBookingFeasible`をシャードのTypedArray表現に合わせて再実装する。確定済み設計判断4「探索基準時刻＋N分 ≤ 乗車時刻」を実行可能性条件とし、「当日でない未来日の探索では制約なし（締切情報として結果に付与のみ）」を反映する。

```
function evaluateBooking(bookingRules: BookingRuleTable, ruleIdx: number, boardingTimeSec: number, searchTime: SearchTimeContext, serviceDate: number)
    -> { feasible: boolean, deadlineSec?: number, reason?: string }:

    if ruleIdx < 0:
        return { feasible: true }   // pickupBookingRuleIdx未設定 = 予約ルールなし（電話予約等の運用注記のみ、booking_ruleでの制約は課さない）

    type ← bookingRules.bookingType[ruleIdx]

    if type == 0:   // リアルタイム予約: 締切なし
        return { feasible: true }

    if type == 1:
        minutes ← bookingRules.priorNoticeDurationMin[ruleIdx]
        if minutes < 0:   // 寛容パーサのフォールバック（10_GTFS-Flex実装仕様.md 3.4節）: 欠損時は締切不明として通す
            return { feasible: true, reason: "prior_notice_duration_min欠損のため締切判定不可" }

        deadlineSec ← boardingTimeSec - minutes * 60   // サービス日内の秒表現。乗車時刻から逆算

        // 「当日でない未来日の探索では制約なし」の判定（確定済み設計判断4）:
        // searchTime.serviceDate（探索基準時刻が属するサービス日）と、
        // このFlexレッグが属するserviceDateが異なれば「未来日探索」とみなし、締切情報のみ付与する。
        if searchTime.serviceDate != serviceDate:
            return { feasible: true, deadlineSec, reason: "締切情報のみ（当日探索ではないため実行可能性には反映しない）" }

        // 当日探索: searchTime.nowSec（探索基準時刻の当日サービス秒）が締切を過ぎていないかを判定
        if searchTime.nowSec > deadlineSec:
            return { feasible: false, deadlineSec, reason: `締切(${formatTime(deadlineSec)})を過ぎているため予約不可` }

        return { feasible: true, deadlineSec }

    if type == 2:
        deadlineSec ← computeType2Deadline(bookingRules, ruleIdx, boardingTimeSec, serviceDate)   // last_day + last_time から日時を計算
        if searchTime.serviceDate != serviceDate:
            return { feasible: true, deadlineSec, reason: "締切情報のみ" }
        if searchTime.nowDateTimeSec > deadlineSec:
            return { feasible: false, deadlineSec, reason: "prior_notice_last_dayの締切を過ぎている" }
        return { feasible: true, deadlineSec }

    return { feasible: true }   // bookingType未定義の異常系（寛容フォールバック）
```

`SearchTimeContext`は「探索を実行している基準時刻」を表す構造体で、`req.searchTime`（未指定時は探索サーバー/クライアントの現在時刻）から`plan()`のエントリポイントで構築する（8章の`PlanRequest`には直接持たないが、実行コンテキストとして内部で保持する）。「当日か否か」の判定は`searchTime.serviceDate`（探索基準時刻をサービス日境界で正規化した日付）とFlexレッグが属する`serviceDate`（シャードの`serviceDate`フィールド、2.2節）の一致比較で行う。

### 4.5 計算量

- `roundMarked`のサイズを`m`、1stopが属する平均Flexグループ数を`g`（瑞穂町は`g=1`だが一般に複数グループ所属を許容、2.2節）、1グループの平均所属stop数を`s`とすると、1ラウンドのFlexスキャインは`O(m × g × s)`。
- 瑞穂町規模（`s=120`）では問題にならないが、都道府県シャードで1グループが数千stopを持つ極端なケースでは`s`が支配的になりうる。9.5節で「グループ内総当たり」を避ける最適化オプション（空間インデックスでの近傍限定）を代替案として記載する。
- `pickupRow`/`dropoffRow`のクロス積は瑞穂町では1×1（各FlexTripにつきpickup_only 1行、dropoff_only 1行）だが、一般化のため`O(|pickupRows| × |dropoffRows|)`を計算量に含める。

### 5.2 アクセス（origin → 初期到達可能stop集合）

```
function resolveAccess(origin: LatLng | StopId, shard: RouterShard, walkLimit: number) -> AccessLegSet:
    if origin is StopId:
        stopIdx ← resolveStopId(shard, origin)
        return { reachableStops: [(stopIdx, 0)] }

    // 座標指定: 空間インデックス（9.3節）でwalkLimit以内の候補stopを絞り込む
    candidates ← spatialIndex.queryWithinRadius(origin.lat, origin.lon, walkLimit)
    reachableStops ← []
    for stopIdx in candidates:
        d ← haversineMeters(origin.lat, origin.lon, shard.stopLat[stopIdx], shard.stopLon[stopIdx])
        if d <= walkLimit:
            reachableStops.push((stopIdx, walkDurationSec(d)))
    return { reachableStops }
```

空間インデックスでの絞り込みは`walkLimit`（既定800m）に一定のマージンを持たせた矩形/円で候補を取得し、その後Haversineで正確な距離を再計算して確定判定する（グリッドインデックスの解像度がHaversine精度に影響しないようにするため、9.3節）。

### 5.3 乗換（transfer）: 事前計算済みテーブルの使用

確定済み設計判断3「transferはシャード内の事前計算テーブルを使用」に従い、ラウンド処理内では**新たに距離計算を行わない**。2.2節の`transfersStart`/`transfersTo`/`transfersDurationSec`は`apps/pipeline`のビルド時に全stop対全stopの候補ペアから`walkLimit`以内のものだけを抜き出して事前計算済み（walkDurationSec関数を`apps/pipeline`側でも共有利用してビルド時に一度だけ計算する）。

```
function relaxTransfers(shard, roundMarked, state, k):
    // 徒歩transferは「乗換回数を消費しない緩和」として扱う（同一ラウンド内の到達時刻改善のみ）。
    // 標準RAPTORのfootpath緩和ステップに相当。
    additionalMarked ← empty set
    for stopIdx in roundMarked:
        for i in shard.transfersStart[stopIdx] .. shard.transfersStart[stopIdx+1] - 1:
            targetStopIdx ← shard.transfersTo[i]
            candidateArrival ← state.τ[stopIdx] + shard.transfersDurationSec[i]
            if candidateArrival < state.τ[targetStopIdx]:
                state.τ[targetStopIdx] ← candidateArrival
                state.bestArrival[targetStopIdx] ← min(state.bestArrival[targetStopIdx], candidateArrival)
                additionalMarked.add(targetStopIdx)
                state.journeyLegs[targetStopIdx] ← { kind: "walk", from: stopIdx, to: targetStopIdx, durationSec: shard.transfersDurationSec[i] }
    roundMarked.addAll(additionalMarked)
```

**乗換回数を消費しない設計の理由**: 標準RAPTORでは徒歩乗換（footpath）はラウンド番号を消費しない「同一ラウンド内の緩和」として扱うのが通例であり、これにより「乗換回数」の直感的な意味（実際に別の車両に乗り直した回数）と一致する。Flexレッグは10_GTFS-Flex実装仕様.md 3.5節の指示通り「1回の乗換」として数えるため、徒歩transferとは異なる扱いになる点に注意（4.1節で再確認）。

## 6. Pareto枝刈りと結果の再構成

### 6.1 Pareto最適性の保証方法

RAPTORはラウンドを重ねるごとに「乗換回数k以内での最早到着時刻」を計算するため、**ラウンドごとのスナップショットを保持すれば、乗換回数×到着時刻のPareto集合が自然に得られる**（乗換回数が少ないほど一般に到着時刻は同じか遅くなるトレードオフの関係）。

```
function reconstructItineraries(shard, state, req) -> Itinerary[]:
    paretoCandidates ← []
    for k in 0..maxK:
        egressArrival ← bestEgressArrival(state.τ_k[k], req.destination)   // 5.4節のEgressLegSetを使い、目的地への最終到着時刻を算出
        if egressArrival is finite:
            paretoCandidates.push({ transfers: k, arrival: egressArrival, snapshotRound: k })

    // 支配関係の除去: 乗換回数がk1<k2かつ到着時刻がarrival(k1)<=arrival(k2)ならk2のレコードは冗長（劣勢解）
    paretoFront ← removeDominated(paretoCandidates)   // 乗換回数昇順に見て、到着時刻が改善しない候補を除去する単純なスイープでO(K log K)

    itineraries ← []
    for candidate in paretoFront:
        legs ← traceLegs(state.journeyLegs, candidate.snapshotRound, req)   // 後述
        itineraries.push(buildItinerary(legs, candidate))

    return itineraries
```

**`τ_k`のスナップショット保持**: 3.3節の`run()`は単一の`state.τ`を破壊的に更新するため、Pareto集合の再構成にはラウンドごとのスナップショットが必要になる。実装上は`state.τ`をラウンド境界で複製する代わりに、`state.journeyLegs[stopIdx]`が既に`round: k`を記録している（3.3節・4.2節の`journeyLegs`代入部分）ため、目的地に到達した`journeyLegs`エントリの`round`を遡ることでPareto集合の各点がどのラウンドに属するかを復元できる。したがって`τ_k`の全ラウンド分の配列複製は行わず、**`journeyLegs`のround番号を唯一の正とする**設計にする（メモリ効率のため、9.2節の方針と整合）。

### 6.2 レッグ列の復元（trace）

`journeyLegs`は「あるStopIdxに、直前どのレッグで到達したか」を記録する連想配列である。目的地からorigin方向へ逆順にたどることでレッグ列を復元する。

```
function traceLegs(journeyLegs, targetRound, req) -> LegRecord[]:
    legs ← []
    cursorStopIdx ← egressBoardingStopIdx(req.destination, targetRound)   // egress徒歩の起点となったstop
    remainingRounds ← targetRound

    while cursorStopIdx has a journeyLegs entry:
        leg ← journeyLegs[cursorStopIdx]
        legs.unshift(leg)   // 先頭に追加（逆順復元のため）

        if leg.kind == "transit" or leg.kind == "flex":
            remainingRounds ← remainingRounds - 1
            cursorStopIdx ← leg.boardStopIdx ?? leg.fromStopIdx
        else:   // "walk"（access/egress/transfer）
            cursorStopIdx ← leg.from   // 徒歩は乗換回数を消費しないため round を減らさない（5.3節）

        if remainingRounds < 0: break   // 安全弁（本来到達しないはずだが無限ループ防止）

    return legs
```

この関数は「乗換k回のPareto点」ごとに独立して呼ぶため、Pareto集合のサイズを`|paretoFront|`とすると再構成全体は`O(|paretoFront| × 平均レッグ数)`で完了する。`|paretoFront| <= maxTransfers + 1`なので、経路数の増加が計算量に与える影響は小さい。

### 6.3 レッグの種類とサマリー算出

```
function buildItinerary(legs: LegRecord[], candidate) -> Itinerary:
    summaryDurationSec ← candidate.arrival - req.departureTime
    transferCount ← count of legs where kind in {"transit", "flex"}  minus 1 (乗り継ぎの回数。最初の乗車はtransferではない)
    requiresBooking ← any(leg.kind == "flex" for leg in legs)

    return {
        legs: legs.map(toPublicLeg),   // 8章のLeg型に変換（StopIdx -> stopId文字列解決含む、2.4節）
        summary: {
            durationSec: summaryDurationSec,
            transferCount: max(0, transferCount),
            arrivalTime: candidate.arrival,
            departureTime: req.departureTime,
            requiresBooking,
        },
    }
```

確定済み設計判断4「結果のレッグには booking情報（電話番号、締切時刻、message）を必ず添付」に従い、`toPublicLeg`は`kind: "flex"`のレッグに対して2.5節の`BookingRuleTable`から以下を解決して添付する。

## 7. isochroneの実装方針

### 7.1 位置づけ

確定済み設計判断5のAPI`isochrone(origin, departureTime, cutoffs[]): GeoJSON`は、`plan()`と同じRAPTORエンジンを再利用し、目的地を指定しない「到達可能性の全stop一括計算」として実装する。自治体向け副産物・審査デモ用（`06_合体案`のアピール構成）という位置づけに対応する。

### 7.2 実装方針: 目的地なしRAPTORの実行

```
function isochrone(shard: RouterShard, origin: LatLng | StopId, departureTime: number, cutoffs: number[]) -> GeoJSON:
    state ← initialize(shard, resolveAccess(origin, shard, walkLimitForIsochrone), departureTime)

    maxCutoffSec ← max(cutoffs)
    maxK ← DEFAULT_MAX_TRANSFERS   // plan()と同じ既定値、または呼び出し側で調整可能にする

    for k in 1..maxK:
        if state.markedStops.isEmpty(): break
        roundMarked ← empty set

        Q ← collectRoutesServing(shard, state.markedStops)
        state.τ_prev ← copy of state.τ
        state.markedStops.clear()
        for (routeIdx, boardFromSeqIdx) in Q:
            // targetPrune の代わりに maxCutoffSec を枝刈り上限として使う（目的地がないため）
            scanRoute(shard, routeIdx, boardFromSeqIdx, state, k, departureTime + maxCutoffSec, roundMarked)

        scanFlexLegs(shard, roundMarked, state, k, isochroneReq(departureTime), roundMarked)
        relaxTransfers(shard, roundMarked, state, k)

        state.markedStops ← roundMarked

    // 全stopの最終到達時刻(state.bestArrival)から、cutoffごとにポリゴンを生成
    return buildIsochronePolygons(shard, state.bestArrival, departureTime, cutoffs)
```

**`plan()`との差分は3点のみ**: (1) 目的地が存在しないため`targetPrune`を`maxCutoffSec`という固定の枝刈り上限に置き換える、(2) `reconstructItineraries`（6章）を呼ばず、代わりに`state.bestArrival`（全stopの最終到達時刻の配列）をそのまま出力に使う、(3) `journeyLegs`によるレッグ復元は不要（到達可否と到達時刻のみが関心対象）。RAPTORのラウンド処理本体（3章・4章）は`plan()`と完全に共有する。

### 7.3 ポリゴン生成

```
function buildIsochronePolygons(shard, bestArrival, departureTime, cutoffs) -> GeoJSON:
    features ← []
    for cutoffSec in cutoffs.sort(ascending):
        reachableStopIdxs ← [stopIdx for stopIdx in 0..shard.stopCount-1
                              if bestArrival[stopIdx] - departureTime <= cutoffSec]
        // 到達可能stop群の凸包（ConvexHull）または α-shape でポリゴン化。
        // v1は実装単純性を優先し凸包を既定とする（凹んだ実際の到達圏より広めに出るが、
        // 自治体向け副産物としては「概観の把握」用途で十分と判断）。
        polygon ← convexHull(reachableStopIdxs.map(idx => [shard.stopLon[idx], shard.stopLat[idx]]))
        features.push({ type: "Feature", properties: { cutoffSec }, geometry: polygon })
    return { type: "FeatureCollection", features }
```

**Flexカバレッジの表現**: Flexグループ内の停留所は、通常の固定路線同様「到達可能なstop」の1つとして`bestArrival`に反映される（4章のスキャンがFlex経由の到達も`state.bestArrival`に書き込むため、7.2節の共有ロジックで自然に処理される）。Flexサービスエリア全体（グループがカバーする面）を明示的にポリゴンとして描く（例:「グループの全stopを結ぶ領域を到達圏として塗る」）機能は、v1では「グループに属する個々のstopが到達可能点として現れる」ことに留め、面としての可視化強化は11章の代替案・未決事項に記載する。

 * 実行前提: originとdestinationが属する地域シャード（＋必要なら鉄道バックボーンシャード）が
 * 既にロード済みであること（呼び出し側、11_アーキテクチャ設計.md 3章(a)/(b)の責務）。
 */
export function plan(req: PlanRequest): Itinerary[];

/**
 * 到達圏算出。cutoffsで指定した秒数ごとの到達可能領域をGeoJSONで返す（自治体向け副産物・審査デモ用）。
 */
export function isochrone(
  origin: LocationRef,
  departureTime: number,
  cutoffs: number[],
): IsochroneResult;

/**
 * シャードのロード（Worker/Node双方で共有するエントリポイント）。
 * plan()/isochrone()を呼ぶ前に、対象地域のシャードを読み込んでおく必要がある(9.4節)。
 * 入力は12_データパイプライン設計.md 4.2節の`Shard`型（JSON.parse済みオブジェクト）。
 * loadShard内部で9.2節のTypedArray変換（RouterShardの構築）を行う。
 */
export function loadShard(shardJson: Shard): RouterShardHandle;

/** loadShardが返す不透明ハンドル。plan()/isochrone()の内部実装が必要とする実データ（RouterShard、2章）を保持する */
export type RouterShardHandle = unknown;
```

**設計上の注記**: `plan()`/`isochrone()`のシグネチャは確定済み設計判断5の記述（`plan(req: PlanRequest): Itinerary[]`、`isochrone(origin, departureTime, cutoffs[]): GeoJSON`）に忠実に従っているが、実行にはロード済みシャードへの参照が必要である。この参照をどう渡すか（`req`に含める、モジュールレベルのシングルトンとして持つ、`RouterShardHandle`を明示的な第一引数にする等）は実装の詳細であり、確定済み設計判断5の型定義自体を変更しない範囲でのAPI設計判断として11章に委ねる。本節では最小限、`loadShard`という前提関数の存在のみ明記する。`Shard`型自体は`packages/router`が定義するのではなく`packages/types`（`11_アーキテクチャ設計.md` 2章の依存方向ルール）または`12_データパイプライン設計.md`が規定する契約を`packages/types`経由で共有する想定であり、`packages/router`はこれを`import type`するのみで自ら再定義しない。

## 9. 性能設計

### 9.1 モジュール構成とホットパスの分離

3章・4章・5章のラウンド処理（`run()`のループ本体）が性能要件のホットパスである。以下を徹底する。

- ラウンド処理内部は文字列比較・オブジェクト生成を行わない。`StopIdx`/`RouteIdx`/`TripIdx`は全て`number`（TypedArrayの添字）として扱い、文字列ID（`stopIds[idx]`等）への変換は6章のレッグ復元（探索終了後、1回だけ）でのみ発生させる。
- `haversineMeters`（5.1節）と4.3節の`estimateFlexDuration`が呼ぶ距離計算は同一実装を共有し、V8等のJITが単一の関数として最適化しやすい形にする（インライン化を妨げるクロージャ多用を避ける）。
- `journeyLegs`（3.2節）はStopIdx添字の配列（`LegRecord | undefined`の固定長配列）として持ち、`Map`オブジェクトを使わない。GCプレッシャの低減（TypedArrayではないが、固定長配列は`Map`よりアクセスが高速でヒープ断片化が少ない）。

### 9.2 データ構造の局所性・TypedArray活用方針

**シャードファイルの形式と、ルーティングコアが実際に読むランタイム表現は明確に分離する（2.1節）。** `12_データパイプライン設計.md` 4章の確定済み設計判断4「v1は圧縮JSON、性能不足が判明した時点でバイナリ化を検討する」により、`fetch`/読み込み対象のシャードファイル自体はv1では列指向JSON（`CompressedStops`/`CompressedRoutes`/`CompressedStopTimes`/`FlexData`/`CompressedTransfers`、`number[]`中心）である。本節が規定するのは、`loadShard()`（9.4節）がこのJSONをパースした**直後**に行う、`RouterShard`（2.2節・2.5節）へのTypedArray変換である。

- **変換処理**: `loadShard()`は`JSON.parse`で得た`number[]`（例: `CompressedStopTimes.stopIdx`, `arrivalSec`）を`new Int32Array(plainArray)`のように1回コピーしてTypedArrayへ変換し、同時にroute単位のCSRオフセット配列（`routeStopsStart`/`routeStops`等）を構築する。この変換はシャードロード時に1回だけ発生するコストであり、探索1クエリあたりのホットパス（3章・4章のラウンド処理）には影響しない。ゼロコピー（`ArrayBuffer`からの直接ビュー構築）はv1のJSON形式では原理的に行えない（コピーを伴う変換になる）が、9.4節が示す通りシャードは一度ロードすればプロセス内に保持され続けるため、変換コストは初回ロード時のみで償却される。
- **理由と効果（変換後の`RouterShard`について）**:
  - **メモリ局所性**: CSR配列表現（`routeStopsStart`/`routeStops`等）は、あるrouteの停留所列を走査する際に連続したメモリ領域を読むため、CPUキャッシュのヒット率が通常の配列オブジェクト（`Array<number>`、要素ごとにポインタ参照が発生しうる）より高くなる。
  - **メモリ使用量の予測可能性**: `Int32Array`は要素あたり4バイト固定。都道府県シャード規模（stop数を仮に1万、trip数を仮に5万、平均trip長10停留所と想定した場合、`stopTimesArrival`+`stopTimesDeparture`で`5万×10×4バイト×2 ≒ 4MB`程度）でも見積もりが容易であり、性能目標「メモリはシャード当たり100MB以内」（確定済み設計判断6）の予算管理に直結する。変換前のJSON文字列表現（`number[]`のJSオブジェクト配列、要素ごとに8バイト以上のボックス化コストがかかる）よりも変換後のメモリ使用量は小さくなるため、100MB予算は主に変換後のTypedArray常駐分を基準に見積もってよい。
  - **GC負荷の低減**: ホットパス内で新たなオブジェクト・配列を生成しない（9.1節）ことと合わせて、探索1クエリあたりのGCプレッシャを最小化する。
- `12_データパイプライン設計.md` 4.4節が明記する通り、Flexの`estimatedDurationSec`はシャードファイルにも事前計算値として格納されない（緯度経度と`DurationEstimator`パラメータのみ格納し、実行時に都度算出する遅延評価方式）。この方針は本書4.3節の`estimateFlexDuration`（実行時に毎回Haversine計算する設計）と完全に一致しており、シャード側・ルーター側の両方で「遅延評価によるシャードサイズ・事前計算コストの抑制」という思想が共有されている。
- `messages`/`phoneNumbers`等の表示用文字列（2.5節`BookingRuleTable`）はTypedArray化せず通常の配列で保持する。ホットパス（3章・4章）で参照されない（6.4節の結果復元時にのみアクセスする）ため、最適化の対象外として割り切る。`12_データパイプライン設計.md`の`CompressedBookingRules`（4.4節）も同様に文字列フィールドを`(string | null)[]`のまま持っており、この点でもシャード形式とランタイム表現の扱いは一致する。

### 9.3 空間インデックス（座標→最寄り停留所解決用）

5.2節の`resolveAccess`・5.4節の`resolveEgress`・7章のisochroneのorigin解決で使う座標→stop近傍探索は、シャードロード時（9.4節のロード処理内）に一度だけ構築する。

- v1推奨: 一様グリッドインデックス（緯度経度を固定サイズのセルに分割し、セル単位でstopIdxのリストを持つ）。KD-treeより実装が単純で、都道府県シャード規模（stop数が万単位程度）でも十分な性能が出る。セルサイズは`walkLimit`の既定値（800m）を基準に、1セルの辺が概ね800m〜1.6km相当になる緯度経度差を採用する。
- グリッド自体もTypedArrayで表現可能（セルごとのstopIdx列をCSR形式で持つ）だが、構築はシャードロード時の前処理であり、ラウンド処理のホットパスではないため、9.2節ほど厳格なTypedArray化を必須としない（実装コストとのバランスで判断してよい）。

### 9.4 シャード遅延ロード

`11_アーキテクチャ設計.md` 3章(a)/(b)が定める「検索地点に応じて必要なシャードのみfetch」という方針を`packages/router`側で受け止める設計。

- `loadShard(shardJson: Shard): RouterShardHandle`（8章。`Shard`は`12_データパイプライン設計.md` 4.2節の型）は1シャード分のパース済みJSONオブジェクトを受け取り、9.2節の変換処理でTypedArray中心の`RouterShard`を構築する。`packages/router`自身はfetchを行わない（環境非依存の原則、`11_アーキテクチャ設計.md` 2章「ブラウザとNode両対応」）。fetchと`JSON.parse`自体は呼び出し側（`apps/web`のWorker起動処理、または`apps/mcp`のリクエストハンドラ）の責務とし、`loadShard()`はパース済みオブジェクトを受け取るところから始まる（8章のシグネチャは`ArrayBuffer`ではなくパース済みJSON型を受け取る形に統一する）。将来`12_データパイプライン設計.md`の未決事項（バイナリ形式への移行、同書8章）が実行された場合は、`loadShard()`の入力型のみを差し替え、9.2節の変換ロジックとその後の`RouterShard`構造・3章以降のラウンド処理は変更しない設計とする。
- 複数シャード（出発地・目的地が異なる地域シャードに属する場合の両シャード＋鉄道バックボーンシャード、`11_アーキテクチャ設計.md` 3章(a)）を扱う場合、`packages/router`は「複数の`RouterShardHandle`を受け取り、結合済みの1つの探索空間として扱う」機能を持つ必要がある。v1の実装方針としては、シャード境界をまたぐstopId（鉄道駅等、複数シャードに重複して存在する接続点）をキーにマージする処理を`loadShard`呼び出し後の追加ステップ（`mergeShards(handles: RouterShardHandle[]): RouterShardHandle`のような関数、8章のAPI定義には含めない内部実装）として持つ。この結合ロジックの具体設計（重複stopの名寄せキー、CSR配列の再構築コスト）は`12_データパイプライン設計.md`とのインターフェース調整が必要であり、11章の未決事項とする。
- 一度ロードしたシャードはWorker/MCPサーバーのプロセス内メモリに保持し、同一シャードへの再アクセスでは再ロードしない（キャッシュ済みデータでの100ms目標、確定済み設計判断6の前提）。キャッシュの破棄タイミング（Workerのライフサイクル、Cloud Runインスタンスのコールドスタート境界）はアプリ側の責務。

### 9.5 性能目標の分解と支配的コスト

確定済み設計判断6「都道府県シャード規模で1クエリ100ms以内（Worker内、キャッシュ済みデータ）」を分解する。

| 処理 | 見積りの考え方 |
|---|---|
| ラウンド処理（3章・4章・5章） | `O(K × R)`（3.3節）＋`O(K × m × g × s)`（4.5節）が支配的。都道府県シャードで`K=6`、`R`（1ラウンドの固定路線走査総量）が万単位でも、TypedArrayベースの単純な数値演算ループであれば現代のJSエンジンで数ms〜数十msオーダーに収まる想定。 |
| Flexスキャン（4.5節） | `s`（1グループの平均所属stop数）が支配的要因。瑞穂町規模（`s=120`）なら無視できる。都道府県シャードで大規模なFlexグループ（`s`が数千）が存在する場合、9.5節の代替案（空間インデックスでの近傍限定、11章）が必要になる可能性がある。**v1では瑞穂町的な規模（`s`が数百程度まで）を前提とし、`s`が性能を支配する規模のデータが実際に来た場合は11章の対策を追加実装する**という段階的な方針にする。 |
| Pareto再構成（6章） | `O(\|paretoFront\| × 平均レッグ数)`（6.2節）。`maxTransfers`が既定6程度なので無視できるオーダー。 |
| isochrone（7章） | ラウンド処理は`plan()`と共有（コストは同等）。凸包計算（7.3節）はstop数に対して`O(n log n)`だが、n（到達可能stop数）は都道府県シャード全体のstop数を超えないため許容範囲。 |

**100ms目標に対する主要リスク**: 4.5節・9.5節が指摘するFlexグループの`s`（所属stop数）が大きいケース。v1のスコープ（1.1節）では瑞穂町規模のFlexデータを前提にしており、この前提が崩れる規模のデータが来た場合の対策は11章に委ねる。

### 10.4 プロパティテスト

確定済み設計判断7「プロパティテスト（三角不等式的健全性、Pareto性）」に対応する。

11. **T-PROP-01（三角不等式的健全性）**: 任意の3停留所A, B, Cについて、`plan(A→C)`の最良到着時刻が`plan(A→B)`経由`plan(B→C)`を連結した到着時刻を上回らないこと（`τ(A→C) <= τ(A→B) + τ(B→C)`相当の性質。RAPTORは最短経路探索なのでこれが破れることは実装バグを意味する）。ランダムに生成した合成フィクスチャ（stop数10〜50、trip数を変動させたもの）で多数の3点組をサンプリングして検証する（プロパティベーステスト、`fast-check`等のライブラリ利用を想定）。
12. **T-PROP-02（Pareto集合の非支配性）**: `plan()`が返す`Itinerary[]`の任意の2要素について、一方が他方を(到着時刻, 乗換回数)の両方で以上に優れることがない（支配関係がない）こと。ランダムフィクスチャで多数回検証する。
13. **T-PROP-03（Flex推定関数の対称性）**: `estimateFlexDuration(A, B) == estimateFlexDuration(B, A)`（Haversine距離自体が対称なため、乗降バッファも対称のはず。実装ミスで非対称になっていないかの検証）。
14. **T-PROP-04（Flex推定関数の三角不等式）**: 任意のグループ内3点A, B, Cについて`estimateFlexDuration(A,C) <= estimateFlexDuration(A,B) + estimateFlexDuration(B,C)`が保証されるか（Haversine距離自体は三角不等式を満たすが、乗降バッファ5分の加算により厳密な三角不等式は成立しない可能性がある。これは実装バグではなく既知の特性であるため、**このテストは「成立しないことの確認」として書く**、11章に注記）。
15. **T-PROP-05（出発時刻の単調性）**: 同一の`origin`/`destination`/`maxTransfers`で、`departureTime`を後ろにずらした場合、最良到着時刻も単調に後ろ、または同じにしかならないこと（早い出発が遅い出発より到着が遅くなることはない）。

### 10.5 API契約テスト

16. **T-API-01**: `plan()`に`departureTime`未指定の`PlanRequest`を渡すとエラーを返すこと（1.1節NS-1の非スコープ表明の実装確認）。
17. **T-API-02**: `Itinerary.summary.requiresBooking`が、legsに`kind: "flex"`が1つでも含まれる場合`true`になること（6.3節）。
18. **T-API-03**: `FlexLeg.booking.message`が`booking_rules.txt`の`message`列の文字列（読点・電話番号含む）とバイト一致すること（6.4節、案内文の完全性）。
19. **T-API-04**: `isochrone()`の戻り値が有効なGeoJSON `FeatureCollection`（`type`, `features`, 各featureの`geometry.type`が`Polygon`または`MultiPolygon`）であること。
