# CLAUDE.md — ノリシロ

このファイルはClaude Code（VSCode）がこのリポジトリで作業する際に常に読み込む前提ファイルである。ここに書かれた制約・規約は他のいかなる指示より優先する。

## プロジェクト概要

「ノリシロ」は、鉄道・バスの固定路線データとGTFS-Flex（デマンド交通）を統合し、「予約が必要な移動手段」を予約締切逆算表示つきで案内する経路検索サービスである。公共交通オープンデータチャレンジ2026（主催: 公共交通オープンデータ協議会・国土交通省、応募締切2027年1月11日23:59 JST）への応募作品であり、交通空白地域の高齢者・その家族・観光客が「乗換アプリに出てこない移動手段」を発見できるようにすることを目的とする。運用コストは0円設計（サーバーを持たず全てCDN配信の静的ファイル、探索計算はクライアント側またはCloud Run上でLLM推論を伴わずに実行）を貫く。

## よく使うコマンド

```bash
pnpm install          # 依存関係インストール（リポジトリルートで実行）
pnpm run typecheck     # 全パッケージのTypeScript型チェック
pnpm run lint          # ESLint（flat config）
pnpm run test          # 全パッケージのvitestテスト実行
pnpm run dev           # apps/web の開発サーバー起動
pnpm --filter @norishiro/router run test   # 特定パッケージのみテスト実行（例: router）
```

PRを出す前に`typecheck`・`lint`・`test`の3つが全て通ることを確認する（CIも同じ3つを実行する、`.github/workflows/ci.yml`）。

## アーキテクチャ要点

詳細は`docs/11_アーキテクチャ設計.md`を正とする。要点のみ再掲する。

- **モノレポ構成と責務**:
  - `packages/types`: 共有型定義のみ。実行コードを持たない。
  - `packages/gtfs`: GTFS/GTFS-Flexパーサ。Raw層（全列optional）とNormalized層（型変換・検証済み）の2層構成。
  - `packages/router`: RAPTOR＋Flex仮想レッグ拡張。ブラウザ・Node両対応の環境非依存な純粋TypeScript。
  - `apps/web`: Vite+React+MapLibre GL JS。探索はWeb Worker内で`packages/router`を実行し、サーバーに探索リクエストを送らない。
  - `apps/mcp`: MCPサーバー（Cloud Run）。同じ`packages/router`をNode環境で実行する。探索アルゴリズムを二重実装しない。
  - `apps/pipeline`: データ取り込み・グラフビルド・地域シャード生成。GitHub Actionsから起動。

- **依存方向ルール（厳格）**:
  - `apps/*`は`packages/*`に依存できる（逆方向は禁止）。
  - `packages/*`同士は`packages/types`以外への相互依存を**禁止**する。特に**`packages/router`は`packages/gtfs`に依存しない**。GTFSデータをグラフに変換する処理（グルーイング）は`apps/pipeline`が担う。
  - この依存ルールに違反する`import`を書かないこと。違反が必要に見える場合は設計側の問題であり、まず`docs/11_アーキテクチャ設計.md` 2章を読み直す。

- **シャードJSON契約は`docs/12_データパイプライン設計.md` 4章が正**である。`packages/router`が読み込むシャード形式（列指向・配列インデックス参照方式、都道府県シャード3MB以下・バックボーン2MB以下）を変更する場合は、この節の契約を変更する提案として扱い、影響範囲（`apps/pipeline`のshard生成ステージ、`packages/router`の`loadShard()`、`apps/mcp`のシャードキャッシュ）を必ず洗い出す。

## コーディング規約

- **TypeScript strict、`any`禁止**。型が不明な場合は`unknown`を使い、型ガードで絞り込む。GTFSの生データのように「実データが仕様から逸脱する可能性がある」場合でも、Raw層で全フィールドをoptionalにする（`docs/10_GTFS-Flex実装仕様.md` 4.3節）ことで対処し、`any`で握り潰さない。
- **`packages/router`は純粋関数・環境非依存**を維持する。`fetch`・`localStorage`・DOM API・Node固有API（`fs`等）への依存を持たせない。ブラウザのWeb Worker内実行とCloud Run（Node）上の実行の両方で同一コードが動くことがこのパッケージの存在意義そのものである。
- **パーサはRaw/Normalized2層**とする（`docs/10_GTFS-Flex実装仕様.md` 4.3節）。Raw層は全列optional・値は文字列のまま。Normalized層は型変換・検証・欠損補完済み。実装コード本体（RAPTOR拡張ロジック等）はNormalized層またはそれ以降の型だけを見て安全に処理できるようにし、「欠損があるかもしれない」という条件分岐を実装コード側に持たせない。
- **エラーは分類型**にする。例外を握り潰して`undefined`を返す、または闇雲に`throw`するのではなく、`docs/12_データパイプライン設計.md` 3章（`ok` / `ok_with_warnings` / `stale` / `skipped`）や`docs/14_MCPサーバーAPI仕様.md` 5章（`INVALID_INPUT` / `DATA_NOT_AVAILABLE` / `SHARD_FETCH_FAILED` / `RATE_LIMITED`）のような、各層が既に定義している分類済みエラー型を使う。新しい層でエラー分類が必要になった場合も、既存の分類方針（「フィード単位で隔離し、1件の破損で全体を止めない」「利用者向けメッセージと開発者向け詳細を分離する」）を踏襲する。
- **寛容パーサ方針を尊重する**: 実データが仕様から逸脱していても、致命的な例外を投げず最大限動作を継続する（`docs/10_GTFS-Flex実装仕様.md` 4章）。新しいパースロジックを書く際、「値が無ければ例外」という実装は避け、フォールバックルールの優先順位（`docs/10` 4.3節）に従う。

## テスト方針

- **瑞穂町フィクスチャ（`packages/gtfs/tests/fixtures/mizuho/`の12ファイル）が正**である。GTFS-Flexパーサ・ルーティングエンジンのテストは、まずこのフィクスチャで検証する。フィクスチャ自体を安易に書き換えない（改変が必要なテスト、例えば寛容性検証テストは、フィクスチャのコピーを改変した別データとして作る。`docs/10_GTFS-Flex実装仕様.md` 5.1節のT-P07〜T-P09を参照）。
- **新機能はテスト先行（TDD）**。`docs/10_GTFS-Flex実装仕様.md` 5章、`docs/13_ルーティングエンジン設計.md` 10章に列挙されているテストケース（T-P*, T-R*, T-V*, T-RAPTOR-*, T-FLEX-*, T-PROP-*, T-API-*）は実装より先にテストコードとして書く。
- 各docsのテストケースには具体的な実測値（例: 殿ケ谷会館〜みずほ病院間2005.5m、推定所要時間759.4秒、予約締切09:30等）が明記されている。これらの数値をそのままアサーションに使うこと（独自の値を再計算しない）。

## 絶対制約

以下は例外なく守ること。設計変更が必要と思われる場合も、まずこれらの制約を維持できる代替手段を検討し、それでも不可能な場合のみ人に相談する。

1. **運用コスト0円設計を壊さない**。データベースを持たない、Cloud Runはmin-instances=0＋max-instances設定必須、外部有償APIを呼ばない（`docs/13_ルーティングエンジン設計.md` 3.2節が既定とするHaversine距離推定を、外部道路網ルーティングAPIに差し替えない）。新しい依存関係・外部サービス連携を追加する前に、それが無料枠内で完結するか確認する。
2. **APIキーをクライアントコードに置かない**。ODPTのAPIキー等は`apps/pipeline`（GitHub Actions内）でのみ使用し、GitHub Secretsで管理する。`apps/web`（ブラウザ）・`apps/mcp`（Cloud Run）のコードにAPIキーを埋め込まない。静的化されたシャードにもAPIキーを含めない。
3. **データ出典クレジット表示を消さない**。フィードごとのライセンス（CC BY 4.0／公共交通オープンデータ基本ライセンス／チャレンジ限定ライセンス）に応じたクレジット表記（`feeds.yaml`の`credit`ブロック、`credits.json`、UI上の「データ出典」ページ）を削除・非表示にしない。チャレンジ限定データも含めてクレジット一覧に表示し続ける。
4. **「予約完了」と誤認させる表現禁止**。Flexレッグ（デマンド交通）の予約はv1スコープ外であり、本サービスは予約の実行を代行しない。予約案内パネルには必ず「この画面から予約は完了しません。電話（または案内先サイト）で直接お申し込みください」（`docs/15_UIUX要件定義.md` 4.3節）に相当する文言を表示する。この文言を省略・弱める言い換えをしない。
5. **チャレンジ限定データを外部再配布しない**。応募規約（`docs/01_開催概要_募集要項.md`）が定める「チャレンジ用データの目的外利用・第三者提供の禁止」を守る。9自治体分のチャレンジ限定Flexデータを、加工済みシャードの形であっても本サービスの目的（経路検索・案内表示）以外の用途で公開・再配布しない。

## Do / Don't リスト

### Do

- 実装を始める前に、対応するdocsの「前提資料との関係」節を読み、他docsとの整合を確認する。
- 新しいテストケースを追加する際は、既存のdocsが定めるテストケース番号の体系（T-P*, T-RAPTOR-*等）に倣った命名をする。
- 未決事項（各docsの末尾「代替案・未決事項」節）に該当する判断が必要になったら、その未決事項が示す「決定に必要な情報」が揃っているかを確認し、揃っていなければ人に相談する。
- シャード形式・API型定義（`docs/12` 4章、`docs/13` 8章）を変更する場合は、影響を受ける全パッケージ（`packages/router`, `apps/pipeline`, `apps/mcp`, `apps/web`）を洗い出してから着手する。
- パフォーマンス上の懸念に遭遇したら、まず該当docsの性能設計節（`docs/13` 9章）・代替案節（`docs/13` 11章、`docs/12` 8章）に既に検討済みの対策が無いか確認する。

### Don't

- `packages/router`から`packages/gtfs`をimportしない。
- Raw層のパース処理で、値が欠損している列に対して例外を投げない（`undefined`として受け止め、Normalized層で補完する）。
- GTFS-Flexの`booking_rules.txt`等の自由文フィールド（`message`列等）を、案内文生成時に言い換え・要約しない（`docs/14_MCPサーバーAPI仕様.md` 3.4節「そのまま提示することを推奨」）。
- 到着時刻指定の逆探索、GTFS-RT反映、運賃計算等、`docs/13_ルーティングエンジン設計.md` 1.2節が明示する非スコープ項目をv1に含めない。
- Cloud Run（`apps/mcp`）にmax-instances未設定でデプロイしない。
- 瑞穂町フィクスチャ本体（`packages/gtfs/tests/fixtures/mizuho/`の12ファイル）を直接改変しない。

## 課題管理の運用

`docs/17_課題管理.md`が唯一の課題トラッカーである。実装中に新たな不明点・要確認事項・リスクを発見したら、コード内TODOコメントや別ファイルに散らさず、必ずdocs/17の表にID採番して追記する。解決時は状態を更新し、解決済みログに解決内容を記録する。

## 作業スタイル（ユーザー設定）

- コスト最小化を最優先する。定型的で量の多い作業（一括データ変換、大量ファイルの機械的リファクタ、ログ解析等）は、サブエージェント（Taskツール）に安価なモデル（haiku/sonnet）を指定して委譲することを検討する。設計判断・レビュー・docsとの整合確認は上位モデル側で行う。
- 回答・報告は簡潔に。冗長な説明を避ける。
